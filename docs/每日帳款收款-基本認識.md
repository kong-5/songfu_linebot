# 每日帳款收款（司機日收表 / 銷貨單對帳 / 收銀機）— 基本認識

> 這份是「今日帳款收款紀錄」功能的**背景知識與資料源定案**，動手開發前先讀。
> 目標：司機出車前印日收表帶現金出去 → 回來清點 → 每日產生當日銷貨單列表、核算應收/欠款 → 點錢對帳。
> 建立日期：2026-07-12

---

## 一句話定位
把**凌越 ERP 的當日銷貨單＋應收帳款**拉到後台，依**司機（＝路線）**分組，做出
「日收表（出車前）→ 收款點錢（回來後）→ 應收/欠款對帳」的流程。松富有 7～8 個司機。

---

## 銷貨單「兩種單號」的來源（重要）
當日銷貨單會有**兩種單號**，各自在凌越有**自己的流水號**，來源不同：

| 類型 | 樣子 | 來源 | 說明 |
|---|---|---|---|
| **純數字** | 只有號碼（無字母） | **直接打入凌越的銷貨單** | 人員在凌越 ERP 直接手開的銷貨單 |
| **A 開頭** | 前面有一個字母 `A` | **訂單 → 拋轉寺岡（Teraoka）→ 轉回銷貨單** | 打完訂單後拋轉寺岡秤重/POS，中間透過公司網站 `https://admin.178go.com.tw`（登入 `/login?to=/dashboard`）用 **EDI 串接**，回轉成凌越銷貨單 |

- 兩種單號**各有各的流水**，報表/對帳時要分別辨識、不可混號。
- 客戶主檔已有 `teraoka_code`（寺岡代碼）欄位，對應「A 類」這條寺岡 EDI 路線。
- 「A 類」的完整鏈路：本系統/人工下訂單 → 寺岡秤重出單 → `admin.178go.com.tw` EDI → 凌越銷貨單（號碼帶 A）。

> ⚠ 待補：需要各拿**一張實際單號範例**（純數字一張、A 開頭一張），確認凌越裡兩者的欄位/單別（`SP_NO` 格式、是否用不同「資料種類／單別」區分），才能在報表正確分流。

## 資料源：金額/應收在凌越，不在本系統
- 本系統的「訂單」是 **LINE 叫貨單**，只有品項＋數量，**沒有金額、沒有應收、沒有付款狀態**。
  本系統單號 `orders.order_no` = `YYYYMMDD+3碼流水`（自家給號，與凌越單號無關）。
- 真正的**銷貨金額、發票、收款方式**在**凌越銷貨單（資料種類 `0000A1`）**：
  `SP_NO` 單號、`SP_DATE` 日期、`SP_CTNO/SP_CTNAME` 客戶、`SP_TOTAL` 含稅總額、
  `SP_INVOICE` 發票、`SP_REM` 備註（內含「時刻／收銀機號／收款(現金)=金額」）。
- **現成查詢管道**：內網 Windows 那台已為 songyang178go 架了查詢服務 `ly_query_server`（預設 port `8787`，
  標頭 `X-Query-Key` 認證），可用 `GET /sp?date=YYYY-MM-DD[&office=...]` **一次拉當天全部銷貨單＋明細**。
  → 這功能的「當日銷貨單列表」**沿用這支即可**，不必從零接凌越。細節見 `docs/songyang178go-銷貨單查詢-API.md`。

## 應收帳款（待接）
- **凌越應該有「應收帳款」**這個資料種類——需向維運端確認**資料種類代碼＋欄位**（客戶、發票/單號、金額、
  已收、未收餘額、帳齡），並在內網查詢服務 `ly_query_server` 加一支對應端點才能查歷史欠款。
- 「今天要收多少」若只算**當日銷貨單總額**，用上面的 `/sp` 就夠；若要含**歷史賒帳/月結欠款餘額**，
  就必須接凌越的應收帳款資料種類。→ 這是決定資料範圍的關鍵，待確認。

## 司機 ＝ 路線
- 系統目前**沒有「司機」實體**；配送用 `customers.route_line`（路線 1–9，NULL＝自取）。
- 司機**通常跟著路線走**，所以「日收表分組」以 `route_line` 為主鍵即可；
  若之後要顯示司機姓名，再加一份「路線→司機」對照（輕量）。

## 回寫凌越（收款結果）— 看有沒有 API
- 收款/點錢結果要不要**回寫凌越**，取決於**凌越有沒有可寫入應收/收款的 API**——沒有 API 就無從回寫。
- 架構限制：雲端連不到凌越 LAN（air-gap），任何回寫都得走**內網代理 + `X-Writeback-Key`** 兩段式
  （比照現有訂單回寫 `scripts/ly_writeback_bridge.py`）。
- → 待維運端確認凌越是否提供「收款/應收銷帳」寫入 API；有才做回寫，沒有就先只在本系統記錄＋出報表。

## 建議分階段（風險由低到高）
1. **當日銷貨單列表 ＋ 司機日收表**（純讀取）：拉 `/sp` 當天銷貨單，依 `route_line` 分組、分「純數字／A 類」列出、
   加總金額，後台可看、可列印/CSV。風險最低，先確認資料對得上。
2. **收款點錢對帳**：新增本系統資料表記錄「實收金額（逐客戶或總數）」，跟應收對帳算差額。
   一台平板跑一個網頁（客戶清單→逐家輸入實收，或投總數）即可，不需專用 POS。
3. **應收帳款/歷史欠款**：接凌越應收帳款資料種類（待確認代碼），併入日收表。
4. **回寫凌越**（選配）：僅在凌越有收款/銷帳 API 時才做，走內網代理兩段式。

## 已確認欄位對照（2026-07-12 實查凌越）
兩種單號**都在 `0000A1` 銷貨單**裡，靠 `SP_NO` 是否以 `A` 開頭分：純數字＝直打凌越、`A`＝寺岡 EDI 回轉。

**銷貨單主表 `SP_`（每張一筆，金額層，不含明細）**
| 用途 | 欄位 | 備註 |
|---|---|---|
| 單號 | `SP_NO` | `YYYYMMDD`＋流水；A 類前綴 `A` |
| 單據日期 | `SP_DATE` | |
| 客戶編號/名稱 | `SP_CTNO` / `SP_CTNAME` | CTNO 例 `AC10013`，對應客戶主檔 `CT_NO` |
| 含稅總額 | `SP_TOTAL` | ＝應收 |
| **未付（應收未收）** | **`SP_NTNPAY`** | 全額未收時＝`SP_TOTAL` |
| 已付 | `SP_PAY` | |
| 未付旗標 | `SP_NOPAY` | `1`＝未付 |
| 結帳方式（單上） | `SP_FKFS` | **實測為空**，改讀客戶主檔 |
| 業務員 | `SP_SALES` | |
> 凌越銷貨單本身即記「收了沒」（`SP_PAY`/`SP_NTNPAY`/`SP_NOPAY`），可當應收/已收基準。

**客戶主檔 `00000D`（`CT_` 開頭）**
| 用途 | 欄位 | 備註 |
|---|---|---|
| 客戶編號/名稱 | `CT_NO` / `CT_NAME` | |
| **結帳方式** | **`CT_FKFS`** | 現成中文，如 `日結-現金`、`月結` ← 報表「結帳方式」欄來源 |
| 付款方式碼 | `CT_PAYFG` | 代碼（如 `6`） |
| 業務員 | `CT_SALES` | 如 `003` |
| 停用碼 | `CT_STOP` | `1`＝停用 |

## 架構決議（2026-07-12，皆已點頭）
- **早上取單＝內網推上雲**：內網 `scripts/ly_sales_push.py` 查當天銷貨單＋客戶主檔，POST 到雲端
  `/admin/lingyue-writeback/cash-ingest`（`X-Writeback-Key`，與回寫同一把）。雲端「該公司該日全表覆蓋」，
  印報表前再跑一次即反映當下。
- **收款客戶主檔**：以 `SP_CTNO`(＝`CT_NO`) 當鍵存 `cash_customer`；`name/fkfs/sales/stop` 由推送覆蓋，
  `is_cash`（是否收現金，seed 由 `CT_FKFS` 含「現金」推定）、`route_line`（路線/司機）、`note` 為後台人工維護不覆蓋。

## 收款資料原子性（跟錢有關，務必守住）
- **重新取單不覆蓋收款**：`cash-ingest` 只 DELETE+INSERT `cash_sales_doc`＋upsert `cash_customer`＋一個時間戳，**完全不碰**
  `cash_payment`／`cash_payment_alloc`／`cash_check`，整段包在 `db.transaction`。→ 改單後重刷，已收現金不會不見。
- **一單只收一次**：`cash_payment_alloc(icpno, sp_no)` 有唯一索引 `idx_cash_alloc_sp_uniq`。兩人同時收同一單→第二筆 INSERT 失敗、交易回滾、回 409，不會重複入帳（`/cash/collect` 的 `already` 預查只是 UX，真正保證靠唯一約束）。
- **收款/取消都在交易內**：`/cash/collect`、`/cash/collect/undo` 皆 `db.transaction`，中途失敗整批回滾。
- **改單/沖帳**：重新取單該日即反映凌越當下（全表覆蓋、未付狀態更新）；收款紀錄為當下快照，不隨改單變動＝保留「實際收了多少」。
