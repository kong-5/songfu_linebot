# 凌越訂單回寫（雲端 ↔ 內網）串接說明

把後台訂單「回寫／匯入」凌越 ERP，並把凌越配發的**單據號碼**顯示回後台。

## 為什麼要分兩段

凌越是**本機 LAN 版** SOAP API（`http://192.168.4.11/erpapi/erpservice.svc`，私有 IP、不走外網）。
後台跑在 **Cloud Run（外網）**，連不到 192.168.4.x 內網，所以：

```
Cloud Run 後台（出資料 / 收單據號）  ←HTTPS→  內網 agent（Dispatch 那台）  ←LAN→  凌越 LyDataIn
```

- 雲端只負責「**輸出待回寫訂單**」與「**接收凌越單據號並顯示**」。
- 真正呼叫凌越寫入 API（`LyGetPassKey` → 寫入 method）的動作，由**內網那台已在跑每日排程的機器**執行。
- **單據號順編**：凌越端／agent 依該日既有單據往後配號，雲端不指定 DocNo，避免撞號。

## 雲端提供的兩支 API（機器對機器）

認證：HTTP 標頭 `X-Writeback-Key`，需等於 Cloud Run 環境變數 `LINGYUE_WRITEBACK_KEY`。
未設定該環境變數時，端點回 `503`（等於停用）；金鑰不符回 `401`。

### 1) 拉「待回寫」訂單

```
GET /admin/lingyue-writeback/pending?date=YYYY-MM-DD
X-Writeback-Key: <金鑰>
```

`date` 省略時預設今天（台北）。只回該日、未刪除、且**尚未回寫**（`lingyue_written_at` 為空）、有可匯出明細的訂單。

回傳範例：

```json
{
  "date": "2026-06-06",
  "count": 1,
  "orders": [
    {
      "order_id": "o_xxx",
      "order_no": "A001",
      "order_date": "2026/06/06",
      "customer_code": "LY001",
      "customer_name": "測試客戶",
      "doc_remark": "",
      "items": [
        { "product_code": "PLU99", "product_name": "高麗菜", "unit": "公斤", "quantity": 10, "item_note": "" }
      ]
    }
  ]
}
```

欄位對應凌越 Excel 匯出邏輯：
`customer_code` = 凌越編號（HQCustCode，無則退寺岡編號）、`product_code` = 凌越料號（erp_code）。
**agent 端再把這些欄位對映到凌越「寫入 method」的實際參數**（參數名稱待跟廠商確認後在 agent 端補上）。

### 2) 回填凌越單據號

```
POST /admin/lingyue-writeback/callback
X-Writeback-Key: <金鑰>
Content-Type: application/json

{ "results": [ { "order_id": "o_xxx", "doc_no": "LY-2026-0016", "ok": true } ] }
```

- `ok: false` 或缺 `doc_no` 的項目視為失敗，**不會**標記為已回寫（下次 `pending` 會再出現，可重試）。
- 成功者寫入 `orders.lingyue_doc_no` 並記錄 `lingyue_written_at`。

回傳：`{ "updated_count": 1, "failed_count": 0, "updated": [...], "failed": [...] }`

## 後台顯示

- 訂單列表：回寫後出現 `↩單據號` 標記（青色）。
- 訂單明細：標題列顯示「凌越單據號 …」。

## 內網 agent（已寫好：`scripts/ly_writeback_bridge.py`）

橋接程式依賴 `D:\Work\lystk_tool\ly_datain.py` 的 `write_invoice / verify_sp_no / delete_invoice`。
單號 `SP_NO` 由 `ly_datain` 依「當日既有單據最大流水 +1」自動順編 → **天然滿足「順著編入不撞號」**，雲端不指定。

### 雲端 pending 欄位 → 凌越欄位對映（已定案）

| 凌越欄位 | 來源（pending JSON） | 備註 |
|----------|----------------------|------|
| `SP_CTNO`   | `customer_code` | 凌越客戶編號（HQCustCode，無則寺岡碼） |
| `SP_CTNAME` | `customer_name` | |
| `SP_DATE`   | `order_date`    | 已是 `YYYY-MM-DD` |
| `SP_REM`    | `doc_remark`    | |
| `SD_SKNO`   | `product_code`  | 凌越料號（erp_code） |
| `SD_NAME`   | `product_name`  | |
| `SD_UNIT`   | `unit`          | |
| `SD_QTY`    | `quantity`      | |
| `SD_REM`    | `item_note`     | |
| `SD_WHNO`   | （預設**留空**） | 之後在凌越補倉別；若凌越不收空倉別，設 `LY_DEFAULT_WHNO` |
| `SD_PRICE`  | （預設 **0**）   | 叫貨單無價；`SD_STOT` 由 `ly_datain` 自動算 |

### 使用步驟（**務必先測一張再整批**）

```bash
# 0) 設定（環境變數）
set LY_CLOUD_BASE=https://<後台網址>
set LY_WRITEBACK_KEY=<與後台 LINGYUE_WRITEBACK_KEY 相同>

# 1) 試跑：只抓+組單印出，不寫凌越、不回填 → 確認對映正確
python scripts\ly_writeback_bridge.py --date 2026-06-06 --dry-run

# 2) 測一張：寫第一張(標記【API測試請刪除】)→回查→自動刪除；不回填後台
#    用這步確認「倉別留空 + 單價 0」凌越會不會收。加 --keep 可保留進 ERP 自己看。
python scripts\ly_writeback_bridge.py --date 2026-06-06 --test

# 3) 確認 OK 後，正式整批回寫並回填後台單號
python scripts\ly_writeback_bridge.py --date 2026-06-06
```

> **待你用 `--test` 確認的兩件事**：凌越是否接受 **空倉別 SD_WHNO** 與 **單價 0**。
> 若被拒：倉別 → 設 `LY_DEFAULT_WHNO=FN005`（或實際倉別）；單價 → 後續再決定改帶價格表。
> 單張寫入失敗不會中斷整批；失敗者不回填，下次 `pending` 會再出現可重試。

### `--test` 預期輸出（成功時長這樣）

```
▶ 抓取雲端待回寫訂單 date=2026-06-06 ...
  共 3 張訂單

▶ 測試寫入第一張：order_id=ord_xxx  客戶=AC30046 知本鼎皇
  [write_invoice] 第 1/1 筆  SP_NO=A202606060007
    attempt=1  rc='0'  ireno='...A202606060007'
    ✅ 寫入成功  新單號=A202606060007
✅ 測試寫入成功，凌越單號 = A202606060007
  回查 ERP：
    SP_NO        A202606060007
    SP_DATE      2026-06-06 00:00:00
    SP_CTNAME    知本鼎皇
    SP_TOTAL     0
    SP_REM       【API測試請刪除】API匯入
    SP_CHECK     0
  已刪除測試單 A202606060007（LyDataDel rc=0，0=成功）

(test：未回填後台。確認無誤後，拿掉 --test 正式整批回寫。)
```

- 看到 `rc='0'` + 回查得到單據 ＝ **空倉別 + 單價 0 凌越收得進去**，可進到正式整批。
- 若 `rc='-2'（寫入失敗）` 或 RuntimeError ＝ 多半是倉別/單價被拒，依上方提示調 `LY_DEFAULT_WHNO`。
- `SP_TOTAL 0` 是預期的（單價 0）；之後在凌越補價即可。

## 每天自動跑（Windows 工作排程器）

內網那台是 Windows（`D:\Work\lystk_tool`），用內附的 `scripts/ly_writeback_daily.bat`：

1. 編輯 `ly_writeback_daily.bat`，填入 `LY_CLOUD_BASE`、`LY_WRITEBACK_KEY`、Python 路徑。
2. 工作排程器 → 建立基本工作 → 觸發程序選「每日」（例如傍晚收單後）→ 動作「啟動程式」選此 `.bat`。
3. 「開始位置」填 `D:\Work\lystk_tool`，確保 import 得到 `ly_datain.py`。
4. 每次執行會寫一份 log 到 `scripts/logs/ly_writeback_YYYYMMDD.log`，方便事後查核。

> 上線前務必先**手動**跑一次 `--test` 確認凌越收得進去，再交給排程器自動跑。
> 排程只是重複呼叫 `bridge.py`；因為「已回寫」的單不會再出現在 `pending`，重複跑安全、不會重複寫入。

## 設定

Cloud Run 環境變數新增：

```
LINGYUE_WRITEBACK_KEY=<自訂一組長亂碼，內網 agent 與後台共用>
```

未設定時兩支端點都停用（回 503），不影響其他後台功能。
