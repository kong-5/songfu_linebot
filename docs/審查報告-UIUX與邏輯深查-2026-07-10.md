## 執行摘要

**整體定位**：這套系統在核心互動（LINE 群自由文字＋手寫照片 AI 收單）上領先台灣同業，國際上唯一同路線且更成熟的是德國 Choco；中/越雙語盤點是市場空白點。架構拓撲正確（air-gap 長輪詢是教科書級 hybrid agent pattern），文件品質在同規模專案中少見。

**三大結構性短板**：
1. **斷單風險尾巴沒收乾淨**——結單 rebuild 覆寫人工改單、去重表廢掉 LINE redelivery、raw_message 併發遺失更新，三者都直接違反「絕不意外斷單」最高原則。
2. **重複寫入與無人監控**——凌越回寫在「寫入成功、回報失敗」路徑會重複開單；內網代理是無人監控的單點；全系統零主動告警（自己就是 LINE Bot 卻沒把推播用在運維上）。
3. **前台幾個「核心流程直接不可用」級的行動端問題**——iOS 打不出負溫度、越南文鍵盤小數逗號被靜默砍掉、盤點超過 1 小時 token 過期無復原。

**成熟度評分**：資料流設計 A−、後台 UI 7/10、故障恢復 B、監控告警 C。

## 第一部分：核心邏輯 debug（webhook / DB / 併發）

### 1.1【高】結單 rebuild 覆寫人工修改
`rebuildOrderItemsFromOrderSources` 以 raw_message 重新解析並 DELETE+INSERT 全部品項；但改單指令（「改第N項」）被 `isRawMessageNoise` 刻意排除在 raw_message 之外（`line.js:416-425`）。客戶改單、機器人回「已更新」後，30 秒結單 rebuild 把數量默默改回原值。同日客戶再傳任何叫貨訊息會重開 session 掛回同單，下次結單再次覆寫（連後台人工修正一起）。
**修法**：改單軌跡另存（order_item_edits 表或 `manual_edited_at` 旗標），rebuild 後重放；或改單以結構化行寫進 raw_message。

### 1.2【高】去重表廢掉 LINE redelivery
訊息一進來先 INSERT `processed_line_messages`（`line.js:812-827`），之後 Gemini 逾時/DB 瞬斷拋錯只被外層 catch 吞掉；LINE 重送同 message.id 命中去重被略過 → 品項永遠沒進 DB。
**修法**：處理成功才落去重標記（或失敗時 DELETE 回滾）；或改租約式標記（status='processing'＋TTL）。

### 1.3【高】worker 端點認證選填
`LINE_WORKER_SECRET` 未設時任何人可 POST 偽造 LINE event 繞過簽章直接建單/標客訴/燒 Gemini 額度。
**修法**：未設 secret 直接 503（比照 `LINGYUE_WRITEBACK_KEY` 的做法）；比較改 `crypto.timingSafeEqual`。

### 1.4【高】raw_message 遺失更新＋session 單實例假設 vs Cloud Tasks
append 是 SELECT→串接→UPDATE（`line.js:427-442`），Cloud Tasks 平行投遞或多實例下兩則訊息互相蓋寫，叫貨行永久遺失；`collectingByGroup`/`autoFinalizeTimers` 是程序內 Map，實例 B 查不到實例 A 的 session 會另開 session、雙 rebuild 交錯。
**修法**：append 改單句原子 SQL（`UPDATE orders SET raw_message = COALESCE(raw_message||E'\n','')||$1`）；session 以 `line_collect_sessions` 表為權威；Cloud Tasks 同群組序列化或明文限定 max-instances=1。

### 1.5【中】其他 webhook 問題
- **unsend 刪整張訂單**（`line.js:753-787`）：客戶收回建單那一句，整張 12 項單無聲消失。改為只移除該則內容或軟刪＋通知管理員。
- **order off 閘門擋掉員工指令**（`line.js:1016-1019` 在員工偵測之前）：內部群員工打「選單」沒反應、員工訊息不進稽核——與 CLAUDE.md「仍回應指令」的描述不符。員工偵測移到閘門前。
- **重啟時 stale session 直接丟棄不補結單**（`line.js:80-84`）：漏 rebuild、漏空籃、漏摘要。stale 改為補跑 finalize（不推播）再刪；部署確認 CPU always-allocated。
- **同日訂單無 `(customer_id, order_date)` 唯一約束**（`line.js:1701-1719`）：併發下同客戶同日兩張單。建部分唯一索引。
- 低：`getTaipeiOrderDate()` 依賴主機 TZ=UTC（`liff/index.js` 的 `stkTaipeiDate` 才是正確寫法，應統一）；`ORDER BY order_no` 文字排序在單日 >999 張時漂移；`processed_line_messages` 無清理；`#盤點` 不吃全形「＃」。

### 1.6 資料庫雙寫
- 【佐證 OK】schema.sql 與 schema.pg.sql 表清單完全一致；alters 兩邊對齊；`INSERT OR REPLACE` 無漏網。
- 【中】`sqlForPg` 的 `?`→`$n` 是無視字串字面值的全域替換——SQL 內含字面 `?` 即參數錯位。目前未發現受害語句，建議 replace 前 assert 參數數一致。
- 【中】Supabase 連線 `ssl: { rejectUnauthorized: false }`（`db/index.js:605-610`）——不驗憑證。改帶 Supabase CA。
- 【低】group_features DDL `feat_stocktake DEFAULT 1` 與「盤點預設關」語意相反；docstring/DDL 註解「三項全開」已過時。DDL default 改 0＋更新註解。

### 1.7 盤點提交（與 LIFF 報告交叉確認）
`liff/index.js:167-182`：DELETE 舊 session/counts → INSERT，無 transaction、`idx_stk_session_date` 非 UNIQUE、整包覆蓋無版本檢查、`body.date` 只驗格式（可覆寫任意歷史日，盤差稽核可被竄改）、逐筆 INSERT 在 Cloud SQL 上 100+ 品項要 100+ 次往返。
**修法**：UNIQUE(wh_code, count_date)＋包 transaction＋bulk insert＋樂觀鎖版本號（不符回 409）＋日期限今日±1。

### 1.8 值得肯定
`order-intent-gate` 安全預設（失敗一律當訂單，絕不錯殺）；凌越 /wait 原子認領租約；SQL 全參數化；LIFF token 走官方 verify 並驗 aud。

## 第二部分：凌越 ERP 整合架構

### 2.1 重複寫入尾巴（高）
agent 寫入凌越成功但 `post_callback` 失敗時 results 只在記憶體，90 秒租約到期後同單被重撿再寫一次。且 90 秒租約短於最壞寫入時間（一批 20 張、每張 SOAP 可達 30-60 秒）。撞號防護（seq reset guard）縮小但未消除 TOCTOU 窗口——重查號到 LyDataAdd 之間櫃台手打單拿走同號會被靜默覆蓋。
**修法**：agent 端 results 落地 journal（callback 成功才刪、啟動重送）；callback 以 order_id 冪等防護；租約延長至 10-15 分鐘或 heartbeat 續租、單批降 3-5 張；寫入後 `verify_or_no` 回查比對內容。

### 2.2 可觀測性（高）
全系統零主動告警：agent 離線、凌越當機、訂單三振出局（失敗 3 次靜默退出佇列）、庫存 18 小時未更新——都要「有人剛好去看」。
**修法**：建「系統告警」LINE 群＋一支 `notifyOps()`，接上四個觸發點：wait 心跳逾時、庫存快照逾時（>26h）、排隊逾 5 分鐘、三振出局。約半天工作量。後台加「凌越同步狀態」總覽頁（佇列中/失敗/最後心跳/快照時間）。

### 2.3 狀態機與一致性（中）
- 「已排隊未回寫」在訂單列表**不可見**（`admin/index.js:9480-9482`）——排隊中的單看起來跟沒轉過一樣。加琥珀色「⏳轉入中」pill，逾 5 分鐘變紅。
- 三振出局無通知、連線類錯誤也計入三振；凌越恢復後不自動重試。區分錯誤類型＋「一鍵重排全部失敗單」。
- 重轉 force 直接清掉舊 `lingyue_doc_no` 無留痕、無對帳。舊單號 append 進歷史欄；每晚用 `0000A0` 回查當日全部回寫單是否仍存在（可複用 txn-wait 骨架）。
- 認領時才組單：按轉入後幾秒內的編輯不會反映到凌越。編輯 API 檢查 `lingyue_queued_at`。
- `erp_txn_res_*` 在 app_settings 永久堆積且拖慢每 1.5 秒的 LIKE 輪詢。加 TTL 清理。

### 2.4 庫存推送（中）
- 快照無版本防倒退：舊快照可能後到覆蓋新資料。交易內比對 `snapshot_at`，較舊回 409。
- **分倉資料閒置**：`SK_NOWQTY` 是公司總量，盤點卻按倉別——多倉共用品項盤差天生失真；凌越 API 本就有 `000009`（逐倉 `@WD_AMT`）已記載未使用。**這是現有已購 API 中價值最高的未開採項。**
- 每日僅 2 次推送（06:00/12:00），傍晚盤點拿早上帳面比實盤。盤點 session 建立時自動觸發刷新（機制現成）；快照早於盤點 2 小時顯示黃色警示。
- 全表覆蓋丟歷史：加輕量 `erp_stock_history`（一天一筆/品項）支撐趨勢與補貨建議。
- 【正面】停用過濾實作正確且自我修復；全表覆蓋已包交易。

### 2.5 前 3 大整合創新提案
1. **分倉庫存（000009）＋盤差半自動回寫**：盤差從名目升級為真實分倉帳面；第二階段用 A1/A2 掛「盤盈虧」虛擬客戶自動開調整單（需會計端認可＋先落地 journal）。
2. **自動補貨建議推播**：庫存快照 × LINE 訂單流速（領先指標）→ 每日補貨清單推採購群。純雲端、不動內網。
3. **整合健康中樞**（心跳＋LINE 告警＋journal）：約 1-2 天投入，是前兩案「敢放手自動跑」的地基，風險回報比全案最高。

- 【正面】群組功能雙入口符合 CLAUDE.md 定案且交叉連結、防呆乾淨。

### 4.4 可及性與效率
ARIA 覆蓋率極低（全檔僅 9 處，多數 modal 無 role/focus trap——建議抽 `sfModal()` helper 統一）；客戶/貨品列表無批次操作、無排序；「品項待對應」統計卡是全庫數字與其他區間卡並排易誤讀。
【正面】⌘K 全站搜尋鍵盤流完整；批次浮動工具列；群發兩段式確認含計費提醒。

### 4.5 創新機會
通訊中心整併（群發＋公告＋行事曆）；統計卡＝篩選器（零後端成本）；「LINE 提醒未盤倉」一鍵追盤（pendingWh 已算好）；訂單審核 j/k 鍵盤流（每日最高頻操作可砍半時間）；儀表板 60s 自動更新。

## 第五部分：市售產品對標研究

### 5.1 最直接對標：Choco / OrderAgent（德國）
與松富同路線（聊天訊息 AI 解析訂單、不要求客戶改習慣），21,000+ 批發商、年 880 萬張訂單。關鍵設計：
- **信心分數＋分流審核**：低信心品項標黃、「原始訊息 vs 解析結果」左右並排核對、按 Accept 才進 ERP。
- **Autopilot**：信心達門檻全自動直進 ERP——自動化是漸進式旋鈕。
- **收單後自動回確認訊息**（closed loop）。
- 通路比松富多「Email＋語音留言」兩個入口。

**松富領先**：LINE 群組場景（台灣真實習慣）Choco 沒有；群組綁客戶＋功能白名單是在地深耕；Choco 不做盤點與物流端。

### 5.2 其他可借鏡
- **Pepper**（美）：品項比對**優先在該客戶歷史訂過的品項內搜尋**（準度大增）；「客戶 X 天沒下單」流失警示。
- **Cut+Dry**（美）：依歷史訂單自動生成「常訂清單」快速下單。
- **TANOMU／クロスオーダー**（日本 LINE 受發注龍頭，10 萬+ 店）：LINE 群發**促銷型錄→點按直接下單**（把收單管道升級成銷售管道）；電子對帳單/出貨明細直接推 LINE；公告一鍵多群發。註：兩者皆是表單/型錄式下單，**無聊天文字 AI 解析——松富在互動成本上領先**。
- **Sortly／Zoho Inventory**：品項照片為中心（對語言不通員工比翻譯更有效）、掃碼跳品項、離線模式、深色模式（冷藏庫）。松富的雙語盤點＋按品項客製輸入欄（效期白名單、中價貨⋯）反而是泛用 App 做不到的。
- **Hybrid ERP 中介**（Oracle/Fivetran）：驗證松富 `/wait` 長輪詢架構就是業界標準 pattern；差的只是心跳監測與冪等鍵明確化。

### 5.3 前 5 大可借鏡功能排序
| # | 功能 | 借鏡自 | 難度 |
|---|------|--------|------|
| 1 | AI 解析信心分數＋低信心標黃＋原文並排審核（通往 Autopilot 的前置） | Choco | 中（數天~一週） |
| 2 | 收單後自動回群組「訂單確認」Flex 卡（糾錯上游化） | Choco/クロスオーダー | 低 |
| 3 | 盤點頁品項照片＋（選配）掃碼跳轉 | Sortly/Zoho | 照片低-中；掃碼中 |
| 4 | LINE 群發促銷型錄→點按即加單 | TANOMU | 中 |
| 5 | 客戶靜默警示（X 天未下單）＋每日出貨明細/對帳推播 | Pepper/Notch | 低-中 |
