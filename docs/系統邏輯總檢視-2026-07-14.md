## 1. 總體評價

系統的**核心防線大多是對的**：庫存推送整批交易、盤點送出「樂觀鎖＋唯一索引＋交易」三保險、
/wait 租約認領、callback 冪等、收款唯一索引防重、後台 auth 覆蓋無漏網、SK_STOP 過濾確實、
group_features 預設值與 CLAUDE.md 定案一致。**踩過的雷（/pending 盲掃、60 張誤寫、#63 盤點覆蓋）
在主路徑上都有防護。**

問題集中在四個型態：

| 型態 | 病灶 |
|---|---|
| **A. 同一條規則有兩份實作，其中一份漂移** | 拆單語意（line.js 存 `''`、後台存 `NULL`）、盤點 API 三胞胎、取號器兩份、掃描引擎兩份 |
| **B. 多公司 icpno 沒貫穿到底** | 盤點唯一索引、分倉快照表、交易快取鍵、照片表、前端草稿鍵都缺 icpno |
| **C. 「查了再寫」沒有 DB 約束兜底** | 拆單建單、收單品項、轉課程按鈕、手動入帳——並發或重試就出重複資料 |
| **D. 修好 A 頁忘了 B 頁** | stocktake.html 的草稿合併/續登/409 重載很完整，scan/freezer-temp/basket-log 都沒跟上 |

## 2. 🔴 高嚴重度（會造成重複出貨、錯帳、資料遺失——建議最優先處理）

### 2.1 拆單：後台路徑會重現「子客戶品項重建回主單→重複出貨」事故
CLAUDE.md 記載的防護（拆單時把當日 NULL 主單標 `''` 桶）**只有 line.js 有做**：
- 後台「依子客戶一鍵拆單」（admin/index.js:16217）與「搬移品項」（:16164）整段交易內**沒有**等效的
  `markSameDayMainOrdersAsSplitBase` 呼叫，來源主單的 `order_sub_split_key` 維持 NULL。
- 後台 `resolveSplitTargetOrder`（admin/index.js:182）新建主客戶桶訂單存 **NULL** 而非 `''`
  （line.js:606 存 `''`）——語意直接和定案矛盾。

失效情境：後台拆單 → 之後按「重新辨識」或 LINE 結單整單 rebuild → NULL＝全部品項 →
子客戶品項重建回主單，與子單**重複出貨**。這正是當初要根除的事故，後台側完整重現。

### 2.2 拆單：並發時同 split key 會開出兩張單（無 DB 約束）
`findOrCreateSplitTargetOrder`（line.js:582-609）與後台版都是 SELECT→INSERT，
`orders` 沒有 (customer_id, order_date, order_sub_split_key) 唯一約束。
Cloud Tasks 兩個 worker 並發處理同子客戶兩則訊息 → 各建一張 → 結單 rebuild 兩張單
各依同一 key 過濾 raw → **品項重複出貨**。另外標 `''` 桶期間並發進來的普通訊息
仍會開出不被標記的 NULL 主單（line.js:571-577 配 2206-2224），窗口變小但沒消失。
建議：部分唯一索引＋撞鍵改重用，這一個不變式能同時收斂 2.1、2.2。

### 2.3 盤點：唯一索引與「啟動去重」缺 icpno——會刪掉另一家公司的整場盤點
- 唯一索引 `idx_stk_session_wh_date_uniq(wh_code, count_date)`（db/index.js:360 / 1103）**無 icpno**，
  但三個送出端都以 (icpno, 倉, 日) 為一場。松富與松揚同倉號（admin:6647 註解自認可能相同）
  同日盤點 → 後送那家永遠 409，該倉當天盤不了。
- 更嚴重：**每次部署啟動都跑**的去重 DELETE（db/index.js:358-359 / 1101-1102）也不比對 icpno——
  若歷史上已存在跨公司同倉號同日兩場盤點，部署當下會**靜默刪除較舊那家的 session＋全部盤點明細**（不可逆）。

### 2.4 分倉快照 `erp_stock_wh_qty` 整張表沒有 icpno
PK 是 (erp_code, wh_code)（db/index.js:281 / 1023）。寫入端（admin:12340-12345）註解假設
「凌越倉號屬單一公司」，與 2.3 的「倉號可能相同」**同一份程式碼兩種前提**。倉號重疊時，
公司 A 推送會刪掉公司 B 同倉號的分倉列；盤點頁「系統量（分倉基準）」會拿到別家公司的量。

### 2.5 掃碼頁（scan.html）兩個會吃掉盤點資料的洞
- **草稿整包覆蓋**（scan.html:297-299）：重演 stocktake 已修掉的 #63——A 掃碼留草稿離開、
  B 用盤點頁送出全倉、A 重開掃碼頁 → B 的資料被 A 舊草稿蓋掉，且 A 拿到新 baseSubmittedAt
  樂觀鎖放行 → 後端 DELETE+INSERT 把 B 的整場盤點清掉。stocktake.html:561-588 已有
  「逐鍵合併＋略過空值」，scan 沒跟進。
- **重送抹掉中貨與效期**（scan.html:657）：續盤把 counted+mid 合併成單一數、送出一律
  `mid:null, expiry:[]`，後端整場刪插 → 盤點頁早上填的中貨拆分與效期批號**全部歸零**
  （合計不變、明細消失，後台「含中貨、效期」欄位憑空不見）。`/admin/scan/items`（admin:8205）
  也不回傳 expiry，同一問題的後端根源。

### 2.6 凌越回寫：「寫入結果不明」＝重複開單，且無告警
ly_agent_gui.py:617-626 對 `write_order` 的**任何**例外（含 SOAP timeout——凌越端可能已開單成功）
都當失敗、不落 journal → 單留在雲端佇列 → 10 分鐘租約到期重派 → **再寫一張**。
雲端 callback 衝突偵測（admin:12228-12235）只能擋「兩次都成功回報且 doc_no 不同」；
第一次根本沒回報 doc_no，第二張正常被接受，第一張成為凌越裡的孤兒重複單、無人知道。
建議：timeout 類例外先用 `verify_or_no`（bridge 已 import）回查凌越當日是否已有該單再定成敗。

### 2.7 /pending 盲掃的「槍」還放在桌上
- `scripts/ly_writeback_daily.bat:40` 不帶參數執行 bridge → 落入 `fetch_pending()` 整批寫入分支——
  **就是 60 張誤寫事故的路徑**，檔頭註解還在教人掛工作排程器。
- 雲端 `GET /lingyue-writeback/pending`（admin:12039）仍上線且**不看 `lingyue_queued_at`**，
  回傳當日全部未回寫單。
- `ly_writeback_bridge.py` CLI 主流程（:381-447）預設就是實寫整批，無 journal、不經 /wait 租約。

CLAUDE.md 的禁令目前只是「約定」，不是「程式保證」。建議：刪 bat；bridge CLI 預設 dry-run、
實寫需明確旗標；/pending 只回 queued 的單。

### 2.8 過時內網腳本仍會被誤用
- `ly_agent_v3.py`：map_order 副本已與權威版漂移（缺 wh_map 備援）、去重只靠記憶體 dict
  （重啟＝同單重寫）、無 first_icpno 防呆。功能 100% 被 GUI 涵蓋。
- `ly_stock_agent.py`：`sys.path` 插 LYSTK_DIR **優先**（:40-41），與 GUI「同層優先」相反——
  會載到 lystk_tool 裡的舊版推送（無 SK_STOP 過濾年代）；配套 bat 仍在引導使用。
- `ly_write_one.py`：雲端零呼叫點、OR_CHECK 固定 0、缺拋轉必備欄位、無撞號 guard。

建議四支（含 bat）直接刪除，見 §6 清理清單。

### 2.9 品項比對 regex 雙重跳脫 bug，直接影響 AI 對品正確率
resolve-product.js:198,223,227 的 `/[\\s,，.。、]+/g` 在 regex literal 中比對的是
「反斜線或**字母 s**」而非空白——`"salsa"` 被洗成 `"al a"`。AI fallback 的候選過濾因此漏配、
掉進前 40 筆隨機池，比對正確率下降。一行修（`\\s`→`\s`），建議連同單元測試一起補。

### 2.10 每日排程的冪等只存在記憶體，且日期用 UTC
- `dist/index.js:335,365`：rhythm 與**每日訂單摘要推播**的「今日已跑」標記是 in-memory。
  Cloud Run 多實例或整點內重啟 → 對客戶群組**重複推播**；scale-to-zero → 可能整天沒觸發。
  應寫進 DB（app_settings）做原子 claim。
- `dist/index.js:205,258`：wholesale/livestock prefetch 預設日期用 `toISOString().slice(0,10)`（UTC）。
  照建議排程台北 06:00/07:00 打時 UTC 還是前一天 → **抓到昨天的行情**。全站已有 taipeiTodayIso 可用。

## 3. 🟡 中嚴重度（依主題）

### 收單/訂單（webhook）
1. **redelivery 品項不冪等**（line.js:1653 等）：租約機制良好，但 `insertParsedItemsForOrder` 無冪等鍵，
   「品項已寫、後續拋錯→刪佔位→重跑」＝品項雙倍。建議品項帶來源 line_message_id 做冪等。
2. **收單寫入全程無交易**（line.js:1641-1704、2354-2367）：建單→附件→品項→raw 全 auto-commit，
   失敗一半可留下「連 rebuild 都救不回的半空單」。db.transaction 存在但 line.js 一處都沒用。
3. **客訴/改單文字會變幽靈品項**（line.js:2239 vs 2265-2327）：intent 偵測**之前**就把文字 append 進
   raw_message → 逐則層面正確不插品項，但結單 rebuild 依 raw 重建時會把「高麗菜5公斤壞掉了」
   解析成品項。intent 是純 regex，移到 append 之前即可。
4. **結單先刪 session 再做事**（line.js:897-899 vs 1044-1046）：rebuild 後空籃/摘要拋錯 →
   空籃沒補、摘要沒發、session 已刪無從重試。刪 session 應移到最後。
5. **拆單通知用 replyToken 且無 push fallback**（line.js:1619→1657、2334→2371）：AI 解析 10-40 秒後
   token 可能已失效，reply 失敗只 log → 客戶不知道被拆單。建議 push-first。

### 後台訂單/收款
6. **轉入凌越重按＝凌越兩張單**（admin:14069）：agent 認領寫入中（doc_no 未回填不觸發 needConfirm）
   再按一次 → 無條件清 claimed_at 重排隊 → 第二條 /wait 再寫。建議租約內未回填時回 needConfirm。
7. **狀態機守衛不對稱**：單筆 approve 有 deleted/complaint 守衛（admin:16786），但 batch-approve
   （:11620）、unapprove（:16863）、客訴轉單（:16917）三條路徑可讓**作廢單復活**→回到 /wait
   回寫候選 → 已在凌越人工刪掉的單可能再次寫入。
8. **「重新取單」與收款分配脫鉤**（admin:12438、13279-13283）：凌越刪單/改額後 `cash_payment_alloc`
   留孤兒、diff 不重算、collect 對已消失單仍插 due=0 的 alloc。
9. **取消收款硬刪三表、無稽核**（admin:13307-13329）：誰何時取消哪筆收款查不到；不存在也回 ok。
   建議軟刪或刪前快照寫 logDataChange。
10. **群組總表過期表單覆蓋**（admin:8497-8523）：儲存時對每個群組整列重寫——A 開著總表、
    B 在客戶編輯改了開關、A 按儲存 → B 的變更被靜默還原。建議只寫有變動列或加樂觀鎖。

### 庫存/盤點

## 7. 挑戰既定原則——需要你決定的討論題

依你的要求，動這些之前先討論，我先給推薦：

1. **「直接改 dist/、無測試」**：不動這個原則也行，但建議底線是——
   (a) 加最輕量的 node:test 鎖住三條踩過雷的規則（subCustomer 清空、split 三態過濾、單位換算優先序）；
   (b) CI 加一個 docker Postgres 冒煙測試（跑 initPg＋核心查詢）。§3-21/22/24 這類
   「本機 SQLite 全綠、雲端 PG 靜默壞掉」的 bug，目前**沒有任何機制能在上線前攔到**。
   成本很低、不改開發流程。**推薦做。**
2. **「盤點誰都可以盤」（stkAuth 只驗 LINE 登入）**：方便性 vs 完整性。目前任何拿到 URL 的
   LINE 帳號可覆蓋盤點、竄改條碼對照、讀全公司庫存。建議折衷：**讀開放、寫（submit/bind）要求
   員工綁定**——員綁機制已存在，只是掛上去。**推薦做**，但這改變員工使用門檻，由你拍板。
3. **「單實例假設」**：line.js 的收單 session、每日推播冪等都建立在單實例上，但 Cloud Tasks
   模式本身就會並發。若短期不打算開多實例，至少把「每日摘要已推」標記落 DB（§2.10）＋
   拆單唯一索引（§2.2）這兩個最痛的先做；其餘 in-memory 狀態可以先立牌子（註解＋文件）。
4. **後台拆單要不要繼承附件/raw**：目前後台複製附件、line.js 不複製，兩者行為不同。
   合併成共用函式時需要定一個標準（我建議：複製附件保留、split key 一律 `''`／子客戶名，
   以 line.js 的語意為準）。
5. **`stocktake_group` 舊表**：行為已不依賴它，但 init 的回填邏輯意味著「刪 group_features 列
   會在重啟時從舊表復活舊設定」。建議擇期把回填改成一次性旗標，舊表純留探索來源。
