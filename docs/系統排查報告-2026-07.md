## A. 立即處理（資安 / 個資外洩）

這幾項牽涉到**憑證外洩**與**權限繞過**，建議優先於一切功能開發。

### A1. 【CRITICAL】正式資料庫密碼寫死在 docs，且已進 git 歷史
- `docs/推Git就部署-設定.md:69` — `DATABASE_URL = postgresql://postgres:hh168888@/songfu?host=/cloudsql/handy-implement-457807-u0:asia-east1:songfu-line-bot`
- 這是**真實密碼＋真實 GCP 專案／執行個體 ID**，不是佔位符。
- **處置**：(1) 立即在 Supabase／Cloud SQL **改密碼**；(2) 把 docs 這行換成佔位符；(3) 用 `git filter-repo`／BFG 清除 git 歷史中的這個字串。單純改工作樹沒用，歷史裡還在。

### A2. 【CRITICAL】後台 session 簽章密鑰有可預測的硬編碼 fallback
- `dist/admin/index.js:1617` — `return process.env.ADMIN_SESSION_SECRET || "songfu-admin-dev-secret-change-in-production";`
- session token 是無狀態的 `base64url({u,exp}) + HMAC(payload, secret)`。若正式環境**沒設** `ADMIN_SESSION_SECRET`，任何人拿原始碼裡的預設密鑰即可自簽任意帳號的 cookie，繞過密碼直接登入為經理。`.env.example` 完全沒提到這個變數，很可能沒設。
- 負責人帳號 `s946185@gmail.com` 硬編碼且恆為 active 經理（`admin/index.js:1645`），偽造目標唾手可得。
- **處置**：確認 Cloud Run 已設 `ADMIN_SESSION_SECRET`；並把程式改成「未設即拒絕啟動」而非用預設值。

### A3. 【CRITICAL】生產個資直接 commit 進 git
- `data/songfu.db`（633 筆客戶）、`cloudsql_export_data_20260323_132005.sql`（632 筆客戶含聯絡人／電話、1999 品項、140 筆訂單明細、LINE group id）。

### A4. 【HIGH】Cloud Tasks worker 端點未設 secret 時完全開放
- `dist/index.js:91-100` — `const secret = process.env.LINE_WORKER_SECRET && ...; if (secret) { 檢查 }`。secret 未設時**跳過整個驗證**，而服務是 `--allow-unauthenticated`。
- 攻擊者可 POST 偽造 event 到 `/api/worker/process-line-event`，憑空建訂單、觸發 Gemini/OCR 花費、對任意 groupId 推訊息 → LINE 簽章驗證形同虛設。
- 同樣模式：`/api/jobs/wholesale-prefetch`（`index.js:173`）、`/api/jobs/rhythm-daily`（`index.js:205`）。
- **處置**：改為「未設 secret 就拒絕」，並用 `crypto.timingSafeEqual` 常數時間比對（現為 `!==`）。

### A5. 【HIGH】LINE_CHANNEL_SECRET 未設時 webhook 不驗簽章仍照常寫入
- `dist/webhook/line.js:1658-1663` — 憑證缺失時掛一個直接 `next()` 的 middleware，不是 fail-closed。任何人可打 `/webhook` 寫訂單。建議缺 config 時回 503。

### A6. 【HIGH】空籃 LIFF API 只驗「是 LINE 使用者」，未驗歸屬
- `dist/liff/index.js:498-617`（`basket-log/lookup-by-group`、`/load`、`/save`）只呼叫 `verifyLineIdToken`，不像 order-review 有 `authenticateLiffEmployee`＋職稱白名單。任何持有效 id_token 者可用任意 `customerId` 讀寫任意客戶空籃、枚舉群組↔客戶對應。
- 另 `basket-log.html:433/445` 把 id_token 放在 **GET query string** → 會進 Cloud Run request log 可重放。

### A7. 【HIGH】備份下載對低權限開放
- `dist/admin/index.js:8339` `/backup/download-json` 無 `requireManager`，任何登入者（含「移工／行政」）可一鍵傾印 customers/orders/order_items/app_settings 全庫。

### A8. 其他（MEDIUM/LOW，防禦縱深）
- 授權模型過寬：作廢／核定／改日期／轉凌越／刪客戶品項只需「登入」，僅「移工」職稱被擋刪除（`admin/index.js:2453,1682`）。
- 無 CSRF token（靠 `SameSite=Lax` 緩解）；session cookie 缺 `Secure` 旗標（`2294`）；密碼最小長度 4（`2272`）。
- Supabase 連線強制 `rejectUnauthorized:false` 關閉 TLS 憑證驗證（`dist/db/index.js:491`）→ 可 MITM。
- 員工綁定碼用 `Math.random()` 非 CSPRNG 且無嘗試次數限制（`employee-line-binding.js:15`＋`webhook/line.js:814`）→ 6 位碼可暴力猜。
- `pg.Pool` 未掛 `error` 事件（`db/index.js:499`）→ idle 連線斷線會 uncaught crash 整個服務。
- Vision API key 放 query string（`vision-ocr.js:68`）、每次呼叫 console.log 完整訂單 JSON（`gemini-order-helpers.js:429`）。

## B. 叫貨邏輯正確性（會直接影響「叫的貨對不對」）

### B1. 【HIGH】重算明細先刪後插且無交易 → 中途失敗整單明細消失
- `dist/lib/rebuild-order-from-sources.js:24-54` `replaceOrderItemsFromParsedRows`：`DELETE FROM order_items WHERE order_id=?` 後逐筆 INSERT，**無 transaction**。中途 crash／並發重算會留下空單或半套明細。
- 同型無交易問題遍布：`admin/index.js` 全檔 grep 不到任何交易；`basket-log.js:106`、`wholesale-snapshot.js:12`（DELETE＋迴圈 INSERT）。

### B2. 【HIGH】多張附件圖任一張解析失敗被默默略過，其他有結果就整單覆寫
- `rebuild-order-from-sources.js:97-118`：某張圖 fetch／解析失敗只 `console.warn` 就 continue；只要其他來源有 ≥1 筆，就整單覆寫 → **失敗那張圖的所有品項默默漏掉**，後台看不到任何「部分失敗」訊號。

### B3. 【HIGH】AI 數量 fallback 鏈失效，無法解析的數量直接歸 0 進單
- `dist/lib/gemini-order-helpers.js:285-287`（`pickQuantityFromRow`）：`coerceQuantityFromGemini` 對無法解析的值回傳 `0`，而 `Number.isFinite(0)` 為真 → 迴圈**第一個非空但解析失敗的鍵就 return 0**，永遠不會 fallback 到後面正確的英文鍵。
- 例：AI 回 `{"數量":"三","quantity":5}` → 得到 0（應為 5）。註解自承是為修「空字串擋英文鍵」而寫，但沒擋「非空但解析失敗」這種情況。

### B4. 【HIGH】非訂單誤判結果被寫入快取 → 真訂單永遠救不回
- `parse-order-from-image.js:222-227` ＋ `non-order-image-detector.js:61-82`：手寫短單「洋蔥3／蒜頭2」（無單位、短、含中文含數字）符合「菜籃標籤照」判別 → 早退回空 items，**且寫入快取**。這與檔尾「空結果不快取」原則矛盾；之後按「重新辨識」永遠命中空快取，除非人工指定旋轉角度繞過。

### B5. 【HIGH】大單 Gemini 輸出截斷 → JSON.parse 失敗 → 整批視覺結果靜默丟棄
- `gemini-order-helpers.js:402,427`：`maxOutputTokens:8192` 且 parse 無截斷防呆。幾十個品項的大單輸出被截斷→parse throw→換模型（同樣截斷）→最終回 null，只剩 OCR 結果，**大單反而最容易漏項**，且無 need_review 標記。

### B6. 【MEDIUM】純文字叫貨的單位判斷只認阿拉伯數字
- `parse-order-message.js:127-131`：`sourceHasExplicitUnit` 用整段訊息判斷、只認「阿拉伯數字＋單位」。
  - 中文數字「豆腐三盒」「青蔥一把」「高麗菜半斤」→ 整段判定「無單位」→ 全部強制當公斤。半斤→0.5 公斤（應 0.3）。
  - 混合訊息中只要有一行寫了單位，其他沒寫單位的行就會用 Gemini 猜的單位而非 fallback（逐行判斷會更準）。

### B7. 【MEDIUM】品名正規化 regex 筆誤
- `resolve-product.js:198,223,227`：`[\\s,，.。、]`（雙反斜線）實際匹配「反斜線」與字母「s」而非空白。同檔第 18 行同功能寫成正確的 `[\s...]`，可見是筆誤 → `cleanForAI`／`normName` 的空白正規化失效，影響 AI fallback 品項對應準確度。

### B8. 【MEDIUM】confidence=1 被誤放大成 100
- `gemini-order-helpers.js:245`：`if (n>0 && n<=1) n=n*100` 無法區分「0-1 量表的 1.0」與「0-100 量表的整數 1」。模型輸出整數 1（幾乎沒信心）→ 變 100 → 讓 Claude 第二意見保險被跳過，最該覆核的那筆反而不覆核。

### B9. 【MEDIUM】模糊匹配自動採用卻標記為高信心
- `resolve-product.js:100-162`：LIKE 模糊匹配排序後直接取 `merged[0]`，回到 `rebuild` 端 `need_review=0`（等同高信心）。模糊命中應標 `need_review` 或降信心，否則「蒜碎/蒜泥」這種近似字會被靜默對錯品項出貨。

### B10. 【MEDIUM】內建質量換算永遠蓋過自訂規則
- `unit-conversion.js:109,199`：台斤/斤/克的內建換算優先於 `line_unit_conversion_rules`，導致 `fromUnits` 含這些單位的自訂規則**永遠不生效**（被默默忽略）。且後台走 `applyUnitConversion`、進單走 `applyOrderUnitConversion`，兩條路徑對 product_unit_specs 的處理不一致，同一張單在不同入口可能換算出不同結果。

### B11. 【MEDIUM/LOW】狀態機與統計口徑不一致
- **核定無狀態機守衛可復活作廢單**：`admin/index.js:11749` approve／`8603` batch-approve 不檢查目前 status，會把 `deleted`/`complaint` 訂單改成 `approved` 重新進出貨/凌越流程。
- **訂單號 race**：`getNextOrderNoAdmin`（`admin/index.js:105`）先讀後寫非原子，同日並發拆單會產生**重複 order_no**；且啟動時若偵測到既有重複就永久跳過建 UNIQUE index（`db/index.js:126`），DB 層保證也失去。
- **凌越重轉**：排隊模式下 `lingyue_doc_no` 被清空的空窗期再點轉入可能**重複排入佇列 → 凌越重複開單**（`admin/index.js:9256,9287`）。
- **各統計模組狀態過濾互相矛盾**：rhythm 不排除 deleted/complaint/voided、war-room 只排除 deleted、customer-scoring／daily-summary 兩者皆排除（`rhythm-analysis.js:92`、`route-war-room.js:91`、`customer-scoring.js:106`、`daily-summary-push.js:118`）→ 戰情室、節奏訊號、客戶評分、每日摘要**同客戶數字對不上**，已刪訂單仍產生缺漏/流失訊號。
- **跨單位直接相加**：`route-war-room.js:95`、`rhythm-analysis.js:96` `SUM(quantity)` 把公斤＋箱＋把混加，量異常訊號可信度存疑。

## C. 可靠性與效能

- **C1【MEDIUM】收單 session 是單實例記憶體狀態**（`webhook/line.js:33` module-level Map），與 Cloud Tasks／多實例衝突 → 同客戶同日開多張單、重複推播。cloudbuild 未設 `--max-instances=1`。
- **C2【MEDIUM】Webhook 先回 200 再背景處理**，Cloud Run 預設 CPU throttling／實例回收下，背景 event、30 秒自動結單、每分鐘 setInterval 排程可能不執行（`webhook/line.js:1700`、`index.js:263`）。部署缺 `--no-cpu-throttling`/`--min-instances`。
- **C3【MEDIUM】重複事件去重不完整**：追加到既有訂單的文字訊息不記 message.id，重送會重複解析寫入；記憶體去重在「處理前」就標記消費，中途拋例外時重送被當重複略過→**訊息永久遺失**；完全沒用 LINE 的 `isRedelivery`/`webhookEventId`（`webhook/line.js:766,94`）。
- **C4【MEDIUM】schema 缺關鍵索引**：`orders(customer_id, order_date)`、`order_items(order_id)`、`order_attachments(order_id)` 皆無索引，webhook 每則訊息都全表掃；`customers.line_group_id` 無 UNIQUE，且每則訊息把全部 customers 撈進 JS 比對（`webhook/line.js:939`）。
- **C5【MEDIUM】後台列表全表載入記憶體篩選**：`/customers`、`/products`、`/api/*-search` 無 SQL WHERE／LIMIT／分頁（`admin/index.js:13212,13871,8360`），逐字搜尋等於每字一次全表掃。
- **C6【MEDIUM】每張訂單圖預設 4~5 次外部 AI 呼叫**（Vision OCR＋auto-orient Gemini＋Gemini 文字＋Gemini 視覺＋低信心 Claude），省錢開關預設關閉，且每張圖重查重組最多 450 個歷史品名塞 prompt（`parse-order-from-image.js`、`auto-orient-image.js`）。**EXIF 方向可用一行 `.rotate()` 免費解決，現在卻花一次 Gemini 呼叫救回**（`line-image-compress.js:12`）。
- **C7【LOW】Few-Shot 範例圖存 /tmp**（Cloud Run ephemeral），重啟後 DB 裡的路徑變死連結，範例庫默默失效、eval 分數被污染（`few-shot-example-save.js:28`）。
- **C8【LOW】客戶收回建單訊息會連坐刪整張訂單**（含之後累加的所有品項），且無稽核記錄（`webhook/line.js:726`）。
- **C9【LOW】內建假日表只到 2027**，2028 起靜默匯入 0 筆（`calendar-holidays.js`）。

## D. 維護性

- `dist/admin/index.js` **單檔 17,062 行**，單一 `createAdminRouter()` 內含約 150 條路由＋整套內嵌 HTML/CSS/JS 字串樣板。跳脫責任散落各處（`escapeHtml` 不跳脫單引號是陷阱）、`nowSql` 方言判斷字串重複 20+ 次、前端 JS 無法 lint/型別檢查。建議逐步拆成 auth middleware＋各領域 router＋view 層＋共用 SQL util。
- SQLite/PG 雙軌靠 `sqlForPg` 正則轉換 SQL（`db/index.js:18`），會誤換字面值裡的 `?`、`INSERT OR REPLACE` 只特判一種句型，屬每個新查詢都要記得的地雷。

## E. 建議新增功能（依效益/成本排序）

1. **客戶叫貨回覆確認**（若尚未有）：結單後把解析結果回傳 LINE 群組，讓客戶「有錯請回覆、逾時視為確認」，把 B 區的解析錯誤在出貨前攔下。（每日摘要已有雛形，可延伸到單筆結單）
2. **解析信心分級與後台高亮**：模糊匹配（B9）、AI fallback、大單截斷（B5）、非訂單誤判（B4）一律標 `need_review` 並在後台紅字提示，讓「靜默失敗」變成「看得見的待辦」。
3. **單位換算 dry-run 測試工具**（後台）：輸入品名＋數量＋單位，即時顯示會怎麼換算，避免 B10 那種「規則被默默忽略」到出貨才發現。
4. **中文數字／半斤支援**（B6）：擴充解析支援「三盒」「半斤」「一打」。
5. **重算改交易化＋失敗即中止**（B1/B2）：任一附件失敗就不覆寫並回報「第幾張圖失敗」。
6. **常訂品項快速下單（LIFF）**：依歷史高頻品項一鍵成單，減少手寫辨識負擔。
7. **回歸測試集擴充**：已有 `gemini-eval-harness`，可把歷史誤判案例沉澱成固定測試集，改 prompt 前跑回歸。
8. **叫貨截止/缺料提醒**：結合 `rhythm-analysis` 的「預期會叫但今天還沒叫」訊號主動提醒業務。
