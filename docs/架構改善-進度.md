# 架構改善 — 進度與檢討

> 依據架構檢討，按「會掉單／算錯單」的嚴重度排序，分小步進行；每步盡量自我驗證後再往下，並定期回頭檢討。
> 分支：`claude/architecture-design-review-JIbAb`

## 背景前提（重要）

- 版控裡目前**只有 `dist/`（編譯後 JS），沒有 TypeScript 原始碼**；`Dockerfile` 直接 `COPY dist`、無 build 步驟。
- 也就是說「實際在跑的就是 `dist/` 本身」。短期內改動直接落在 `dist/`；**「原始碼進版控」列為後段必辦項（D1）**。
- 本機驗證：`npm install --omit=dev` 可成功編譯 `better-sqlite3`，能用 SQLite 路徑啟動與測試（PG 路徑無本機實例，採標準寫法 + 推理驗證）。

## 階段與優先序

### Phase B — 資料一致性（低風險、高價值）
- [x] **B1. DB 層新增 `transaction(fn)`**（SQLite + PG，純新增、不動現有呼叫點）
  - 驗證：SQLite commit/rollback 測試 PASS。
- [x] **B2. `getNextOrderNo` 原子化**（並發不重號、對既有資料相容不跳號）
  - 驗證：SQLite 三情境（預設／起始號／既有資料）+ 真實 wrapper 連號／跨日測試 PASS。
  - ⚠️ 待辦：`orders(order_date, order_no)` 唯一索引「先做重號檢查再加」（既有資料可能已有舊競態造成的重號，盲建會在啟動失敗）。
- [ ] **B3. 一則訊息一個交易**：把 `webhook/line.js` 的各寫入路徑逐一包進 `transaction`（先單一訂單文字路徑試點，再擴及拆單／圖片）
- [ ] **B4. 純邏輯單元測試**：`parse-order-message`、`unit-conversion`、拆單分組

### Phase C — 掉單止血（中風險）
- [ ] **C1. webhook 全面走 Cloud Tasks**（正式環境強制入隊，handler 只驗簽→入隊→回 200）
- [ ] **C2. 收單 session 狀態 DB 化**（取代記憶體 `collectingByGroup`）
- [ ] **C3. 結單改 Cloud Tasks 延遲任務**（取代記憶體 `setTimeout` 30 秒）

### Phase D — 工程體質與安全
- [ ] **D1. TypeScript 原始碼進版控 + CI build**（移除「只 commit dist」的反模式）
- [ ] **D2. admin session secret 強制設定**（未設 `ADMIN_SESSION_SECRET` 即拒絕啟動）
- [ ] **D3. 正式 migration 機制**（取代每次啟動跑的 try/catch ALTER TABLE）
- [ ] **D4. 結構化日誌 + 關鍵指標／告警**（收到訊息數 vs 成功落庫數，避免靜默掉單）
