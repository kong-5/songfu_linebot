# 收單後沒資料、前幾天資料消失：請設定 Cloud SQL（長期記憶）

若您遇到：

- **剛收單，但後台訂單查詢裡沒有這筆**
- **前幾天的訂單／客戶／品項都不見了**

代表目前**沒有使用雲端資料庫**，程式用的是容器裡的 **SQLite**，資料只存在那台機器的磁碟上，**重新部署、關機、縮容或換機器時都會被清掉**，所以無法長期記憶。

---

## 解決方式：改用 PostgreSQL（Cloud SQL）

請讓 Cloud Run 連到 **Cloud SQL（PostgreSQL）**，並設定環境變數 **DATABASE_URL**。設定完成後，所有訂單、客戶、品項都會寫進 Cloud SQL，**長期保留**，不會因為部署或重開而消失。

### 步驟概要

1. **在 GCP 建立 Cloud SQL 執行個體**（若尚未建立）  
   - 資料庫類型選 **PostgreSQL**  
   - 建立一個資料庫（例如 `songfu`）與使用者（例如 `postgres`）並設密碼  

2. **在 Cloud Run 設定**  
   - 到 [Cloud Run](https://console.cloud.google.com/run) → 點您的服務（例如 songfu-line-bot）  
   - **編輯並部署新修訂版本**  
   - **連線**：新增您的 **Cloud SQL 執行個體**（這樣 Cloud Run 才能連到 DB）  
   - **變數與密碼**：新增變數  
     - 名稱：`DATABASE_URL`  
     - 值：`postgresql://postgres:您的密碼@/songfu?host=/cloudsql/專案ID:區域:執行個體ID`  
     - 範例：`postgresql://postgres:MyPass123@/songfu?host=/cloudsql/my-project:asia-east1:songfu-db`  
   - 儲存並部署  

3. **之後**  
   - 每次 `git push` 部署只會更新程式，**不會清掉 Cloud SQL 裡的資料**  
   - 收單的訂單會正常出現在後台，前幾天的資料也會一直保留  

更詳細的連線與權限設定，請參考同資料夾的 **「觸發程式設定-DATABASE_URL」** 或 **「做法二-Cloud-SQL-詳細步驟」**。

---

## 如何確認已在使用 Cloud SQL

- 後台**工作台**首頁若顯示綠框 **「PostgreSQL (Cloud SQL) － 訂單、客戶、品項會長期保留」**，代表已連線，資料會長期記憶。  
- 若顯示紅框 **「資料不會長期保留」**，代表尚未設定 DATABASE_URL 或連線失敗，請依上方步驟設定。

---

## 我記得之前有設定過，但現在還是看不到訂單？

不需要「開啟雲端檔案」；訂單是存在 **Cloud SQL 資料庫** 裡，不是單一檔案。若以前設過現在卻沒資料，請依序檢查：

1. **後台工作台現在顯示綠框還是紅框？**  
   - 若仍是**紅框**「資料不會長期保留」→ 代表目前這版服務**沒有**讀到 DATABASE_URL（可能被覆蓋或沒帶到這台）。  
   - 請到 [Cloud Run](https://console.cloud.google.com/run) → 您的服務 → **編輯並部署新修訂版本** → **變數與密碼**，確認有 **DATABASE_URL** 且值正確（見上方連線字串格式）。儲存後再部署一次。

2. **Cloud SQL 執行個體有沒有在跑？**  
   - 到 [Cloud SQL](https://console.cloud.google.com/sql) 看您的 PostgreSQL 執行個體。  
   - 若狀態是「已停止」，請**啟動**它（Cloud SQL 可設成閒置時關機省錢，但關機期間服務連不到，就會變成用 SQLite、資料看不到）。

3. **Cloud Run 有沒有掛上「連線」？**  
   - 編輯並部署新修訂版本 → 下方 **「連線」** 區塊要**新增 Cloud SQL 連線**，選您的那個執行個體。  
   - 沒掛的話，就算設了 DATABASE_URL 也連不到資料庫。

4. **部署方式會不會把變數蓋掉？**  
   - 若觸發程式用的是 **cloudbuild.yaml** 且裡頭有傳環境變數，每次 push 部署可能會**只帶建置檔裡那幾個變數**，沒帶 DATABASE_URL，所以舊的 DATABASE_URL 會消失。  
   - 建議：觸發程式改用 **cloudbuild-no-secrets.yaml**，所有金鑰與 **DATABASE_URL** 只放在 Cloud Run 的「變數與密碼」裡設一次，之後 push 只更新程式，不會改掉變數。詳見「推Git就部署-設定」做法 B。

照上面檢查並修正後，重新部署一次，再收一筆單測試；若工作台變成綠框，訂單就會長期保留並在訂單查詢裡看到。

---

## 四項都設了還是看不到訂單？再檢查這些

5. **結轉日期是否把訂單篩掉了？**  
   訂單查詢預設只顯示「訂單日期 ≥ 結轉日期」的訂單。若結轉日期被設成**未來**或**很晚的日期**，舊訂單都會被篩掉。  
   - 在訂單查詢頁點 **「顯示全部訂單（不限結轉日期）」**，看是否會出現訂單。  
   - 若點後有訂單：代表資料在 Cloud SQL 裡有，只是被結轉日期篩掉了；請在工作台上方把**結轉日期**改回今天或您要的起始日。  
   - 若點後仍「無訂單」：代表資料庫裡目前真的沒有訂單（見下一點）。

6. **收單時有沒有真的寫入這台 Cloud SQL？**  
   - 確認您開的後台網址是 **Cloud Run 的網址**（例如 `https://xxx.run.app/admin`），不是本機（localhost）。本機會連到本機的 SQLite，看不到雲端資料。  
   - 到 **Cloud Run → 您的服務 → 日誌**，收單時是否有 `[LINE] 進入收單流程`、`訂單已寫入` 等 log；若有錯誤（例如連線逾時、密碼錯誤），會寫在日誌裡。

7. **啟動時有沒有連到 PostgreSQL？**  
   在 Cloud Run 日誌裡看**服務啟動**那一筆，應有 `[startup] 使用 PostgreSQL (Cloud SQL)`。若出現的是 `未設定(SQLite)`，代表當時環境沒有 DATABASE_URL，可能是該修訂版本沒帶到變數。
