# 資料處理規則（權威版）

> 松富 LINE Bot ↔ 凌越 ERP 之間所有資料流的**唯一規則文件**。
> 程式實作以 `scripts/ly_writeback_bridge.py` 的 `map_order()` 為唯一權威——
> 視窗版（`ly_agent_gui.py`）、常駐版（`ly_agent_v3.py`）都呼叫它，**改規則只改一處**。
> 歷史教訓與必備設定見 `凌越回寫-規則與必備設定.md`；環境架設見 `內網設定與部署.md`。

---

## 1. 訂單回寫（雲端 → 凌越訂貨單 0000A0）

### 1.1 什麼時候寫（最重要）

**只寫使用者在網站按過『上傳凌越』的單。**

- 網站點「上傳凌越」→ 後台把該筆排入佇列（`lingyue_queued_at`）。
- 內網代理長連線 `GET /admin/lingyue-writeback/wait` —— **只回佇列中的單**。
- 寫入成功 → `POST /callback` 回填凌越單號 → 該筆退出佇列。

❌ 常駐代理**禁止**使用 `/pending`（當日全部未回寫清單）。
2026-07-08 曾因此誤寫 60 張整天訂單（單號 202607080064–202607080123，已刪）。
`/pending` 只留給 CLI 手動批次（`ly_writeback_bridge.py --date …`）與診斷。

### 1.2 單別

寫入凌越**訂貨單（客戶訂單）＝資料種類 `0000A0`**，用 `ly_order.write_order`。
（不是銷貨單 0000A1；LINE 叫貨屬「訂單」階段，之後在凌越轉銷貨出貨。）
單號 `OR_NO` 由 `ly_order` 依當日流水自動順編，雲端不指定、不會撞號。

### 1.3 主表欄位對映

| 凌越欄位 | 來源 | 規則 |
|---|---|---|
| `OR_CTNO` | customer_code | 凌越客戶編號（HQCustCode，無則寺岡碼） |
| `OR_CTNAME` | customer_name | |
| `OR_DATE1` / `OR_DATE2` | order_date | 正規化 `YYYY-MM-DD`（斜線轉橫線） |
| `OR_REM` | doc_remark | 測試單前綴【API測試請刪除】；拋轉會寫 `#` 進備註 |
| `OR_CHECK` | 設定 | **預設 `1`＝已審核（拋轉需要）**；`0`＝未審核（好刪，但拋轉抓不到） |
| `OR_MAKER` | 設定 | 製單人＝操作員代碼（預設 `052`，非 LY）——比照手打單 |
| `OR_CREATEDATE` / `OR_CREATENAME` | 自動 | 寫入當下時間＋操作員代碼。**拋轉依建立日期抓單，缺了拋不過** |
| `OR_MODIFYDATE` / `OR_MODIFYNAME` | 自動 | 比照手打單帶入 |
| `OR_FKFS` | 客戶主檔 `CT_FKFS` | 付款方式。凌越畫面選客戶會自動帶，**API 寫入要自己補** |
| `OR_SALES` | 客戶主檔 `CT_SALES` | 業務員，同上 |

### 1.4 明細欄位對映

| 凌越欄位 | 來源 | 規則 |
|---|---|---|
| `OD_SKNO` | product_code | 凌越料號（erp_code）。**空料號→整行跳過並警示**（凌越不收），其餘品項照常寫 |
| `OD_NAME` | product_name | |
| `OD_UNIT` | 貨品主檔 `SK_UNIT` **優先** | 查不到才用雲端單位＋正規化（見 §1.5）。⚠ 抄雲端「公斤」會讓凌越把 `OD_IS_PACK` 設成論件，寺岡秤重點不了 |
| `OD_WARE` | 貨品主檔 `SK_RKWHNO` **優先** | 查不到→固定倉別 `LY_DEFAULT_WHNO`。**拋轉要扣庫存，空倉別拋不過** |
| `OD_QTY` | quantity | 空值補 0 |
| `OD_PRICE` | 設定 | **留空＝不送**→凌越依客戶售價表自動帶價；只有明確設定才強制覆寫 |
| `OD_REM` | item_note | 有備註才送 |

### 1.5 單位正規化表（退路用）

| 雲端單位 | 寫入凌越 |
|---|---|
| 公斤 / kg / KG | **KG** |
| （空白） | KG |
| 其他（把、箱、盒、台斤…） | 原樣送出 |

對照表：`ly_writeback_bridge.py` 的 `UNIT_MAP`。要擴充改那張表。

### 1.6 失敗與重試語意

- 單張寫入失敗**不影響**其他張；失敗者不回填 → 仍留在佇列，下次可重試。
- `callback` 只把 `ok=true 且有 doc_no` 的標成已回寫；其餘保留。
- 已回寫的單不會再出現在佇列/待回寫 → 重複跑安全、不重複寫入。
- 同一張排隊單已寫過（agent 記住 `order_id|queued_at`）→ 只補回填、不重寫。

### 1.7 重轉（已回寫的單要重寫）

網站該筆按「↩ 重新轉入」→ 後台清掉舊單號並重新排入佇列 → 代理重寫。
⚠ 重轉會在凌越**產生新單**，舊單不會自動刪，要先進凌越刪舊單。

## 2. 庫存推送（凌越 → 雲端）

| 項目 | 規則 |
|---|---|
| 資料來源 | 貨品主檔 `000000`：`SK_NO`/`SK_NAME`/`SK_SPEC`/`SK_UNIT`/`SK_NOWQTY`(目前庫存)/`SK_RKWHNO`(倉別) |
| 上行端點 | `POST /admin/lingyue-writeback/inventory-push`（全表覆蓋 `erp_stock_items`） |
| 即時觸發 | 長連線 `GET /inventory-wait`；後台按「庫存更新」→ 立刻撈凌越推送 |
| 定時 | 預設每天 06:00、12:00 各推一次＋程式啟動推一次 |
| 數值處理 | 千分位逗號移除；整數化（`10.0`→`10`）；無效值→0 |

## 2b. 庫存頁點品項查進銷存（單品項，on-demand）

在「庫存管理 → 目前庫存」點任一品項 → 右側抽屜顯示該品項的凌越貨品主檔資料
（目前庫存/單位/規格/倉別等「重點」＋「凌越全部欄位」摺疊）。

- **為什麼經內網小幫手**：凌越在內網，雲端連不到 → 點擊只是「排一筆請求」，
  由內網代理長連線 `txn-wait` 撿走、查凌越、`txn-callback` 回填，網站再顯示。
- **負擔很小**：點了才查、一次一筆；快取 5 分鐘（同品項 5 分內再點用快取，不重打凌越）。
- **資料來源**：貨品主檔 `000000` 單筆查詢（一次 SOAP，輕）。
- **重點欄位是「猜測名」**：`ly_stock_push.py` 的 `SUMMARY_FIELDS` 列了常見進銷存欄名
  （SK_INQTY/SK_OUTQTY/SK_LASTINDATE…），凌越沒有的自動略過；**全部欄位一律附上**。
  👉 實際跑一次看「凌越全部欄位」，把真正的進/銷/存欄名補進 `SUMMARY_FIELDS` 即可。
- 端點：`txn-wait`（agent 長連線）、`txn-callback`（agent 回填）、
  瀏覽器 `POST /inventory/stock/txn-request`＋`GET /inventory/stock/txn?code=`（用 app_settings 協調，跨實例安全）。

## 3. 認證與端點速查

機器對機器一律帶標頭 `X-Writeback-Key`＝後台環境變數 `LINGYUE_WRITEBACK_KEY`。
後台未設定該變數→端點回 503；金鑰錯→401。

| 端點 | 用途 | 常駐代理用？ |
|---|---|---|
| `GET  /admin/lingyue-writeback/wait` | 「上傳凌越」佇列（長連線） | ✅ 回寫唯一入口 |
| `POST /admin/lingyue-writeback/callback` | 回填凌越單號 | ✅ |
| `GET  /admin/lingyue-writeback/inventory-wait` | 庫存即時刷新（長連線） | ✅ 庫存 |
| `POST /admin/lingyue-writeback/inventory-push` | 推目前庫存 | ✅ 庫存 |
| `GET  /admin/lingyue-writeback/pending?date=` | 當日全部未回寫 | ❌ 僅 CLI 手動/測試連線探針 |

任一端點 404 ＝ 線上後台版本較舊、缺該端點 → 需重新部署後台。

## 4. 凌越查詢通則（lystk / ly_order / ly_datain）

- **一定要注入有逾時的 zeep 用戶端**（`Transport(timeout=60)`）——lystk 預設無逾時會卡死。
- 目前庫存掛在**貨品主檔 000000 的 `SK_NOWQTY`**，不是庫存種類 000009（000009 只有批號品項）。
- 主檔查詢一律加程序內快取（客戶 `_cust_cache`、貨品 `_prod_cache`），避免重複 SOAP。
- 明細行要另外用 `LyDataOut` 帶 `idetfields` 解析 `LYDATADETAIL`（`verify_*`/`lystk.query` 只回主表）。
- 常用資料種類：`000000` 貨品、`000001` 客戶、`0000A0` 訂貨單（OR_/OD_）、`0000A1` 銷貨單（SP_/SD_）。
- 公司代碼：`00` 松富、`01` 龍港、`03` 桂田（松成門市部門欄位 `SP_DPNO`）。

## 5. 出錯救援

- **刪誤寫的訂貨單**：`ly_writeback_undo.py --range 起 迄`（先核對）→ 加 `--confirm` 才真刪。
- **驗證欄位帶入**：`ly_writeback_bridge.py --test-ctno 客戶代碼`（寫一張測試單印出所有關鍵欄位）。
- **查某張單實際欄位**：`ly_query_unchecked_sales.py --doc 單號 --details`（診斷拋轉缺什麼）。
- 訂單資料在雲端資料庫始終保留；凌越端刪單不影響網站顯示（會顯示已轉入，用「重新轉入」重排）。
