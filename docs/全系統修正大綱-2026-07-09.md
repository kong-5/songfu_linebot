## 批次一【最急】：重複出貨 / 凌越重複開單

### 1-1. 文字收單「一句多子客戶」拆單後，主訂單重複包下全部品項（雙倍出貨）

- **位置**：`dist/webhook/line.js:1739`（`appendRawLineToOrders` 把本則文字寫進主訂單）、`:1845-1868`（多子客戶拆單分支）、`dist/lib/rebuild-order-from-sources.js:159-160`（`order_sub_split_key === null` 時回傳全部解析行）。
- **問題**：session 進行中收到「A校 蘿蔔3斤、B幼 高麗菜2斤」這種一句多子客戶的訊息時，文字先寫進主訂單 raw，拆單分支只「新建」各子客戶訂單、不清主訂單。結單重算時主訂單（split_key=null）把 raw 重新解析成**全部**品項 → 主訂單一份全品項＋各拆單各一份＝重複出貨。
- **為何確定是 bug**：圖片路徑同款問題已修過（`dist/webhook/line.js:1203-1206` 有修正註解：拆單時不把本次內容寫回主訂單），文字路徑漏修。
- **修改辦法**：比照圖片路徑——把「是否多子客戶、要不要拆單」的判斷**提前到寫入主訂單 raw 之前**；若判定拆單，該則文字只寫入各拆單的 raw，不寫主訂單。若判斷無法提前（需先解析），則在拆單分支內把剛 append 進主訂單的那段 raw 移除（需要 `appendRawLineToOrders` 的反向操作，或記住 append 前的 raw 快照後還原）。
- **驗證**：模擬群組訊息一句含兩個子客戶 → 結單後主訂單**不含**該批品項、兩張拆單各含自己的；再驗一般單子客戶／無子客戶訊息行為不變；驗 session 第一句就是多子客戶的情境（主訂單品項應為空）。

### 1-2. 凌越回寫 callback 無防護：租約過期後重複認領 → 凌越兩張單、雲端單號被覆蓋

- **位置**：`dist/admin/index.js:10918-10925`（callback 成功回填的 UPDATE）、`:10824`（`LEASE_MS` 90 秒租約）、`:10843-10847`（/wait 條件式認領，這段本身正確）。
- **問題**：callback 回填只 `UPDATE orders SET lingyue_doc_no=?, lingyue_written_at=? ... WHERE id=?`。agent A 認領後寫凌越超過 90 秒，agent B 重新認領又寫一張（凌越端兩張單）；兩者先後 callback，後到者直接覆蓋 `lingyue_doc_no`，另一張成為無記錄的孤兒單。
- **修改辦法**：
  1. `/wait` 認領時把 `lingyue_claimed_at` 的值一併回傳給 agent 當「認領憑證」；agent callback 時帶回。
  2. callback 的 UPDATE 加兩個條件：`AND lingyue_written_at IS NULL AND lingyue_claimed_at = <憑證>`。
  3. `changes === 0` 時不回填、記 warn log（代表發生了過期重寫，凌越端可能有重複單，要在回應與後台醒目提示，讓人工去凌越刪多的那張）。
  4. 內網端 `scripts/ly_writeback_bridge.py` / `ly_agent_gui.py` 的 callback payload 配合帶上憑證欄位（雲端要向下相容：舊 agent 沒帶憑證時，退回只加 `lingyue_written_at IS NULL` 條件）。
- **驗證**：模擬同一單兩次 callback（第二次帶舊憑證）→ 第二次 changes=0、單號不被覆蓋、有 warn log。

### 1-3. 「重新轉入」會打斷進行中的寫入 → 保證重複開單

- **位置**：`dist/admin/index.js:11253`（重新轉入清 `lingyue_claimed_at`、`lingyue_doc_no`、`lingyue_written_at` 並重排佇列）。
- **問題**：agent 正在寫凌越時按「重新轉入」，租約被清空 → 下一輪 /wait 立刻再發同一張給 agent → 凌越又一張；先前那次的 callback 還會把舊單號回填成「已回寫」。
- **修改辦法**：重新轉入前檢查 `lingyue_claimed_at` 是否仍在租約有效期內（now − claimed_at < LEASE_MS）；是 → 拒絕並回「內網代理正在寫入這張單，請等它完成或失敗後再重轉」；否 → 照常重排。搭配 1-2 的憑證機制，舊 callback 也不會再污染。
- **驗證**：把某單 `lingyue_claimed_at` 設為現在，按重新轉入 → 被拒；設為 5 分鐘前 → 正常重排。

### 1-4. `/pending` 端點沒有佇列條件——誤寫 60 張事故的門還開著

- **位置**：`dist/admin/index.js:10753-10765`。
- **問題**：`/pending` 回傳「當日全部未回寫單」，沒有 `lingyue_queued_at IS NOT NULL` 條件、也不排除 `complaint`。規則說它只留給 CLI 手動診斷，但程式層面任何持金鑰的呼叫都能拿到整天的單——2026-07-08 誤寫 60 張走的正是這條路。
- **修改辦法**：預設加上 `AND o.lingyue_queued_at IS NOT NULL AND COALESCE(LOWER(TRIM(o.status)),'') NOT IN ('deleted','complaint')`；保留診斷需求則加查詢參數 `?include_unqueued=1` 才回全部（回應中標注 `"warning": "含未排隊單，僅供診斷，禁止據此寫入"`）。同步更新 `docs/資料處理規則.md` §1.1 說明。
- **驗證**：不帶參數呼叫 → 只回按過「上傳凌越」的單；帶參數 → 全回且有警語。

## 批次二：錯單 / 漏單

### 2-1. 「今天叫了什麼」項次與「改第N項/刪第N項」基準不一致 → 客戶自助改錯行

- **位置**：`dist/webhook/line.js:1442-1445`（顯示查詢**無 ORDER BY**）vs `:1459-1468`（改/刪單 `ORDER BY oi.id`）。
- **修改辦法**：`:1442` 的查詢加 `ORDER BY oi.id`，與改單、刪單、線上改單（`:1495`）、結單摘要（`:628`）全部對齊。一行修。
- **驗證**：造一張多品項單，確認「今天叫了什麼」列出順序與「線上改單」項次一致。

### 2-2. 訊息去重寫在「處理前」＋逐則吞例外回 200 → 崩潰/AI 失敗即永久漏單

- **位置**：`dist/webhook/line.js:811-826`（入口即寫 `processed_line_messages`）、`:1874-1879`（逐則 catch 吞例外）、`:1911-1935`（Cloud Tasks at-least-once）。
- **問題**：去重紀錄在建單前就落地。之後的圖片下載/OCR/Gemini 要 10–40 秒，若 instance 被回收或 Gemini 噴錯，該 message id 已被標記處理過，重投遞被去重擋掉 → 單永久不建立、幾乎無痕跡。
- **修改辦法**（最小改動、不動整體時序）：
  1. 逐則 catch（`:1874-1879`）內，把該 message id 從 `processed_line_messages` **刪除**再記 log——讓 LINE/Cloud Tasks 重投遞可以重跑這一則。
  2. 對「本則已成功建單/併單」的正常路徑不動（去重仍防同 id 重複建單）。
  3. 若要更穩：`processed_line_messages` 加 `status` 欄（processing/done），入口寫 processing、成功後改 done；重投遞遇 processing 且超過 5 分鐘視同失敗允許重跑。此法需 SQLite/PG 雙寫加欄位，工程量較大，可列為二階段。
- **驗證**：在解析途中人為丟例外 → 去重紀錄被清；重送同一訊息 → 成功建單、不重複。

### 2-3. 群組換綁客戶時歸屬不定（first-match 無排序）

- **位置**：`dist/webhook/line.js:1098-1102`（收單歸屬）、`:884-888`（空籃）、`:1015-1016`（盤點內部群）、`:1051`（員工對話）。
- **問題**：`customers.line_group_id` 無唯一性，換綁沒清舊客戶時兩筆 active 客戶同 group_id；`find` 取第一筆且查詢無 ORDER BY（PG heap 順序不穩定）→ 同群組的單時而掛新、時而掛舊客戶。
- **修改辦法**：
  1. **寫入端防呆**（治本）：後台綁定 group_id 時，檢查其他 active 客戶是否已用同一 group_id；有 → 自動把舊客戶的 group_id 清空並在操作結果註明（或擋下要求先解綁，擇一，建議自動清＋註明）。
  2. **讀取端保底**：上述四處查詢加 `ORDER BY updated_at DESC`（取最近綁定者），比對到多筆時記 warn log。
  3. 資料清乾淨後（確認無重複），PG 建部分唯一索引 `CREATE UNIQUE INDEX IF NOT EXISTS ux_customers_line_group ON customers(line_group_id) WHERE line_group_id IS NOT NULL`（比照 `ux_orders_order_no` 的建法：啟動時嘗試建立，失敗不擋啟動）。SQLite 同名索引。
- **驗證**：兩客戶設同 group_id → 後台綁定第二個時舊的被清；群組收單掛到新客戶。

### 2-4. 批次改出貨日不看凌越/匯出狀態 → 雲端與凌越日期永久不一致且無警示

- **位置**：`dist/admin/index.js:10366-10397`（`/batch-change-date`）。
- **修改辦法**：對選中的單先查 `lingyue_written_at / lingyue_doc_no / sheet_exported_at / lingyue_exported_at`；任一非空的單**預設跳過**，回應列出「N 張已上傳凌越/已匯出，未更改：單號…」。若營運需要強制改，加 `force` 參數＋前端二次確認彈窗（文案明講「凌越那張單的日期不會跟著變，需人工進凌越改」）。
- **驗證**：混合選取已回寫與未回寫的單批次改期 → 只有未回寫的被改，回應清單正確。

## 批次三：進凌越/匯出的資料品質（直接影響庫存帳）

### 3-1. 後台改單不套用單位換算，與收單行為不一致

- **位置**：`dist/admin/index.js:13873`（編輯明細時 `applyOrderUnitConversion(db, { rules: [] }, null, ...)` ——傳空規則、無品項），對照收單路徑 `:2598-2645`（`autoConvertOrderItemsToKg` 有套 `product_unit_specs.conversion_kg`）。
- **問題**：收單「2 把」自動換公斤；後台把同品項改成「3 把」則原樣存「把」。同一張單兩種單位並存，上傳凌越時未換算的「把」直接寫入。
- **修改辦法**：編輯儲存時改為與收單同一條換算鏈：載入實際規則（`app_settings.line_unit_conversion_rules`）＋該品項的 `product_unit_specs`，傳入已解析的品項而非 `null`。換算成功 → 依 `remarkStyle` 重寫備註；未觸發換算但舊備註是「原 X 單位」格式 → 清除舊備註（避免「6公斤/備註:原2把」矛盾）。
- **驗證**：對設有 0.2kg/把 的品項，後台改成「3 把」→ 存成 0.6 公斤、備註「原訂3把」；改成「5 公斤」→ 原樣、舊「原訂…」備註被清。

### 3-2. 上傳凌越的品項篩選三個洞：need_review 照傳、0 量照傳、空單位預設「公斤」

- **位置**：`dist/admin/index.js:11127-11162`（buildLingyuePreview）、`:10648-10672`（凌越 xlsx 匯出）、`:10770-10787`（pending）——**三處要同步改**。
- **修改辦法**：
  1. `need_review = 1` 的品項：計入 `missing_count` 一樣的攔截機制（轉入入口 `:11240` 已有缺料號攔截，擴充為「缺料號或待確認品項 > 0 → 擋下並列出品項」），讓使用者先到訂單明細確認完再上傳。
  2. `quantity <= 0` 的行：**先跟老闆確認業務意圖**——空籃/號碼籃品項數量 0 是揀貨與籃子追蹤用，若凌越不需要 0 量明細，三處一律過濾 `qtyNum > 0` 才輸出；若某些 0 量行必須進凌越（不太可能），改為只過濾籃類品項。
  3. 單位空白：不要預設「公斤」（`|| "公斤"`）。改成傳空字串，讓內網橋接端用貨品主檔 `SK_UNIT` 補（`scripts/ly_writeback_bridge.py:216` 本來就是 SK_UNIT 優先，雲端單位只是退路）。
- **驗證**：造一張含 need_review 品項的單按上傳 → 被擋並列名；含 0 量籃品項的單 → 預覽/xlsx/實傳皆不含該行；品項單位清空後上傳 → 凌越端單位為主檔 SK_UNIT。

### 3-3. 舊版每日盤點 `daily_inventory` 整包覆蓋＋先查後寫 → 兩人同倉同日互相蓋掉

- **位置**：`dist/admin/index.js:6834-6843`。
- **修改辦法**：改成 UPSERT＋合併：
  1. id 本來就是 `recordDate_warehouseId` 組合鍵 → 用 `INSERT ... ON CONFLICT(id) DO UPDATE`（PG 與 SQLite 語法皆支援）消除撞主鍵 500。
  2. `items` 不整包覆蓋：UPDATE 分支把新 items 與既有 items 以品項 id 為 key 合併（新提交的品項覆蓋同品項、保留他人已填品項）。合併在應用層做：交易內 `SELECT ... FOR UPDATE`（PG）/ 交易（SQLite）→ merge → UPDATE。
- **驗證**：兩個請求各帶不同品項同時提交同倉同日 → 兩批品項都在；同品項後提交者生效。

### 3-4. 匯出與採購分析不濾作廢單、彙總分裂

- **位置與修法**：
  1. `/export/download` CSV（`dist/admin/index.js:10014-10019`）：加 `AND COALESCE(LOWER(TRIM(o.status)),'') NOT IN ('deleted','complaint')`（比照訂單列表 `:9173`）。
  2. 凌越 xlsx（`:10596-10692`）與揀貨單（`:10445-10508`）：依勾選 ids 查詢時同樣加狀態過濾，且只對實際匯出的單蓋 `lingyue_exported_at`/`sheet_exported_at` 章。
  3. 採購分析 `/admin/logistics/procurement`（`:8628-8639`）：`logistics_order_items` 若有 `voided_at` 欄位則加 `voided_at IS NULL`；GROUP BY 改為 `GROUP BY COALESCE(p.name, oi.raw_name), oi.unit`（同品項不再因「已對照/待對照」分裂成兩列），顯示時已對照者用品名、未對照者標「(待對照) raw_name」。
- **驗證**：含 deleted 單的日期匯出 CSV → 不出現；採購分析同品項只剩一列。

### 3-5. 數量解析：範圍值「2-3斤」被砍成 2、負數不擋

- **位置**：`dist/lib/gemini-order-helpers.js:224-231`（`coerceQuantityFromGemini` 用 `parseFloat`）、`dist/lib/parse-order-message.js:136`、`dist/webhook/line.js:382-384`（insert 前只擋非有限值）。
- **修改辦法**：
  1. coerce 時偵測 `\d+(\.\d+)?\s*[-~～至]\s*\d+(\.\d+)?` 範圍格式：取**上界**（備貨寧多勿少）並把品項標 `need_review=1`、備註記「原文範圍 2-3」。
  2. 負數：clamp 成 0 並標 `need_review=1`（不要無聲吞掉）。
- **驗證**：訊息「白菜 2-3斤」→ 數量 3、need_review、備註含原文；「-5斤」→ 0＋need_review。

## 庫存面實質改善方案（供後續討論，本批不實作）

上面批次一～三修的是「別再把帳弄髒」；下面是把庫存數字**變得可用**的建設性方案，依投入小→大排序：

### I-1. 推送擴充欄位（半天工）
`ly_stock_push.py` 每品項多帶：`SK_SAFEQTY`（安全存量）、`SK_UPDATE1`（最近入庫日）、`SK_UPDATE2`（最近出庫日）；`erp_stock_items` 加對應欄位（雙寫）。這三個欄位是負庫存分類與叫貨表的原料，凌越都現成有、只是沒推上來。

### I-2. 負庫存分類看板（1-2 天工）
現在「負數很多」無法行動，是因為分不出哪些正常。用 I-1 的欄位把負庫存品項分三類顯示在「目前庫存」頁：
- **時間差型**（可忽略）：近 N 日有出貨且歷史上進貨頻繁——銷貨先入帳、進貨晚補的正常負數。
- **疑似分裝型**（要查）：本品長期負、且存在「同名不同規格」的兄弟品項近期出量異常——母件分裝成子件沒做帳的特徵。
- **僵屍型**（要清）：負了 30 天以上、無進出——主檔錯誤或早該調整的帳。
每類給筆數徽章＋一鍵匯出，讓「清負數」變成每週可執行的例行工作，而不是面對一整片紅字。

### I-3. 盤差基準修正（2-3 天工）
- 盤點**送出時**由伺服器重讀當下 `erp_stock_items` 存 `sys_qty`（取代前端凍結值，`dist/liff/index.js:181`）。
- 增加「**合併盤差**」視圖：同一品項若當日在多個倉都有盤，以「各倉實盤合計 vs 公司總量」計盤差（單倉品項行為不變）。這是在「凌越 000009 只出批號品項、蔬果拿不到分倉帳」（`scripts/ly_query_stock.py:14-15` 實測）前提下，最務實的盤差修正。
- 盤差確認後給「差異單」出口：人工核可 → 記錄調整原因 → 產出一張可貼進凌越做調整單的清單（先不自動回寫，降低風險）。

### I-4. 分裝 BOM＋未拋轉偵測（3-5 天工）
- 新表 `product_bom(parent_erp_code, child_erp_code, ratio)`，後台維護「一箱母件 = N 包子件」。
- 每日檢查：子件近 7 日銷量 > 0 且母件庫存未動、或子件庫存持續變負而母件為正 → 在庫存頁與（未來的）叫貨表標「疑似分裝未拋轉」。
- 把「分裝有沒有做帳」從月底盤點才發現，變成隔天早上就看到。

### I-5. 叫貨建議表 v1（1-2 週工，等 I-1~I-3 落地後做）
每日清晨產出一頁（每列一品項）：
`建議叫貨 = 明日需求 − 可用庫存 + 安全庫存(SK_SAFEQTY)`
- 明日需求 = LINE `order_items`(出貨日=明日) ＋ 紙本 `logistics_order_items` 依 SK_NO 彙總，經 `unit-conversion` 正規化到採購單位；對照不到料號的品項獨立列「待對照」警示區。
- 可用庫存 = `erp_stock_items.qty`（負數以 0 計並標記）。
- 附欄：近 4 週同星期平均出量（給無單品項的參考）、負庫存/分裝警示。
- 學校（松揚）需求：先以「該校歷史同星期出量」估算並標「預估（無菜單）」；菜單匯入功能列 v2，屆時預估升級為「菜單×人數」。
