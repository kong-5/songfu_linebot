7 個面向並行地毯式稽核：DB 雙寫一致性、交易原子性與併發、多公司 icpno 連貫性、LINE webhook 收單邏輯、庫存/盤點資料流、安全性、網頁設計一致性。所有發現均附 file:line 證據；三個高嚴重度發現已另行人工抽查確認非誤報。

## 高嚴重度（會遺失資料）

### H1. 盤點送出：樂觀鎖檢查在交易外，併發送出會把先送者整場盤點刪掉且不報 409
- `dist/lib/stocktake-api.js:141-147`（樂觀鎖比對）在交易（`:157-173`）之外執行；交易內是先 DELETE 舊 session+counts 再 INSERT。
- 時序：兩支手機同倉同日、都在對方送出前開頁（baseSubmittedAt 皆 null）幾乎同時按送出。A 先 commit；B 在「檢查已通過、交易未開始」的窗口後進交易——B 的 DELETE 把 A 剛寫入的整場盤點刪掉再插自己的。唯一索引兜不住（列已被刪，INSERT 不衝突），也不會回 409。A 的實盤數整場遺失。
- LIFF、掃碼頁、網頁版盤點三入口共用此函式，皆受影響。
- 修法：把樂觀鎖比對（或 re-check）搬進交易內第一步；一處修好三入口。

### H2. 圖片與觸發詞路徑會把品項插進子客戶拆單訂單 → 結單 rebuild 整批丟掉（漏出貨）
- 文字收單已有 reroute 防呆（`dist/webhook/line.js:2428-2446`），但：
  - `line.js:1706-1719`（收單中、圖片非拆單分支）直接 `insertParsedItemsForOrder(db, session.orderId, ...)`——而 session 可能掛在子單上（拆單建 session 時 `orderId: mainBucketOrderId || touchedOrderIds[0]`，整則訊息都是同一子客戶時 mainBucket 為 null，`line.js:1754`、`1898`）。
  - `line.js:1724`、`1910-1933`（無 session 圖片／觸發詞含品項）的同日單查詢不過濾 `order_sub_split_key`，同日第一張若是子單就插錯桶。
- 情境：群組先傳「整則都是某分店」的訊息（session 掛子單），接著傳一張無子客戶標記的訂單照片 → 品項寫進子單 → 結單 rebuild 依 split key 過濾（`rebuild-order-from-sources.js:292-302`）把這批品項全部丟掉。
- 修法：把 2428-2446 的 reroute 邏輯套用到圖片與觸發詞路徑；同日單查詢加 split key 過濾。

### H3. 後台人工改單不寫 `order_item_edits`，結單 rebuild 會默默還原人工修正
- 改單軌跡重放只保護 LINE 端指令：全 codebase 寫 `order_item_edits` 的只有 `dist/webhook/line.js`（770、1986、2012、684）；`dist/admin/index.js` 0 筆，但後台確實直接改品項（`admin/index.js:2969`、`18031` 等）。
- 情境：員工上午在後台改數量/刪誤辨品項 → 同日客戶又在群組傳訊息 → session 重開掛回同單 → 結單 rebuild「DELETE 全部品項→依 raw 重建→只重放 LINE 軌跡」→ 後台修正被靜默還原。
- 修法：後台改品項的各寫入點同步寫 `order_item_edits`（沿用 `recordOrderItemEdit` 的 match_key 規則）。

## 中嚴重度

### 資料庫層
- **M1. PG `erp_stock_items`/`erp_warehouse` 主鍵遷移非交易＋半途失敗後永不重試**（`db/index.js:1113-1125`、`1172-1185`）：DROP pkey 成功後 ADD PRIMARY KEY 失敗（連線中斷/鎖逾時），下次啟動守門條件為 false → 永遠不重試，表從此無主鍵 → 所有 `ON CONFLICT (icpno, erp_code)` upsert（庫存推送、倉別設定）整批失敗。對照 `erp_stock_wh_qty` 遷移（`:1133-1150`）已包 BEGIN/COMMIT，這兩張沒回補。
- **M2. PG `erp_stock_item_photo.icpno` 的 ALTER（`:924`）排在 CREATE TABLE（`:1152`）之前**：全新 PG 庫首次啟動缺欄 → 照片上傳 500（重啟自癒）。
- **M3. init 用大 try 包多句 DDL＋`catch(_)` 全吞**（`:345-360`、`:1126-1165`、`:386-404`、`:1192-1229`）：單句失敗會靜默跳過同 try 後續建表，執行期才 500 且啟動 log 無訊息。
- **M4. `sqlForPg` 防護缺口**（`:19`、`:25`）：攔不住 `datetime('now','+8 hours')` 這類帶修飾詞寫法（regex 只吃 `datetime('now')`、fail-fast 只查 `date('now'`），未包 isPg 分支的新 code 會原樣送進 PG 執行期才炸。

### 併發／原子性
- **M5. 主客戶桶（split key `''`/`NULL`）check-then-insert 無唯一約束兜底**（`line.js:574-602`、`1724`、`1834`、`1910`；索引定義 `db/index.js:239` 明確排除 NULL 與 ''）：同群兩人幾乎同時各傳一則叫貨（跨 webhook 請求/多實例）→ 各建一張同日主單，第二張變孤兒單。子客戶（非空 key）路徑有唯一索引＋catch 重查，安全；只有主桶是漏洞。
- **M6. 後台「貼上原始訊息重辨識」Gemini 呼叫夾在 read-modify-write 中間**（`admin/index.js:17943-17963`）：解析期間（10-40 秒）客戶在 LINE 補叫貨（原子附加成功）→ 解析完 `UPDATE raw_message = finalRaw` 整欄覆寫把新行洗掉 → 下次 rebuild 永久漏該品項。webhook 端同型問題 2026-07-10 已修（`line.js:738-744`），此後台入口漏修。
- **M7. 結單 finalize 無跨實例互斥**（`line.js:918-924`、`2570`、`948`）：去重僅靠單實例記憶體 Map；部署換版重疊/多實例下兩邊各跑一次 finalize → 客戶收到兩次結單摘要；`insertEmptyBaskets`（`empty-baskets.js:35-38`）無交易無唯一索引，可插出重複空籃品項。可仿 writeback `/wait` 的條件式 UPDATE 認領。
- **M8. 條碼綁定「先刪後插」無交易、撞 PK 無重試**（`liff/index.js:208-209`、`admin/index.js:9261-9262`、`9468-9469`）：兩支手機同時綁同一條碼 → 後者 500「綁定失敗」；DELETE 與 INSERT 間崩潰＝舊綁定遺失。改 `ON CONFLICT DO UPDATE`（`admin/index.js:7095` 已有雙分支示範）。

### 多公司 icpno
- **M9. inventory-push 端點未驗證 icpno 格式**（`admin/index.js:13497`，`String(body.icpno||"00").trim()`；對照 cash-ingest `:13669` 有 normIcpno）：內網 `LY_ICPNO` 誤填 `2`/`0 2` → 整批寫入 icpno='2' 孤兒列，SQL 端 COALESCE 比不到（之後正確推送的 DELETE 也刪不掉），JS 端 normIcpno 卻回退 '00' 把它混進松富報表。入口加 `/^\d{2}$/` 驗證、非法 400。
- **M10. 盤點頁草稿 localStorage 鍵漏 icpno**（`liff/stocktake.html:440` `'stk_draft_'+code+'_'+date`；對照 scan.html:234 已含公司）：凌越倉號可跨公司重複——同機同日先盤松富某倉再切松揚同倉號：草稿互相覆蓋、恢復草稿會把別家公司的數字帶進本倉；「草稿中」徽章也跨公司誤標。`/admin/inventory/entry` 共用同檔一併受影響。

### 庫存／盤點資料流
- **M11. 必盤 fallback 基準尺度不一致**（`stock-mustcount.js:43` vs `:55-59`）：現量取公司總量，fallback 卻拿上次 `stocktake_count.sys_qty`——該值在有分倉資料的倉凍結的是分倉量。公司昨日快照缺（新公司/推送中斷）走 fallback 時，跨倉持有品項差值恆大於門檻 → 整倉大量誤標「必盤」。
- **M12. 盤差％公式不一致、負庫存時符號反轉**：每日盤點頁/CSV 用 `diff/sys`（`admin/index.js:6712-6721`、`7205`）；統計圖表用 `diff/max(|sys|,1)`（`:7424`）；異常排查表用 `diff/max(|latest|,1)`（`:7262`）。庫存可為負（正常）——sys=−5 實盤 0 時每日盤點頁顯示「+3 (−100.0%)」數字與％正負矛盾，且同一筆盤差三處顯示不同。建議統一為 `max(|·|,1)` 分母。

### webhook 收單邏輯
- **M13. `modify_order` intent 誤觸吃掉加購品項**（`line.js:472` pattern 命中「再加3把蔥」，優先序 modify > add，`:434`）：收單中此訊息不寫 raw、不解析 → 品項完全不入單，只剩 remark `[客戶意圖：改訂單]`。
- **M14. 收單關鍵字過寬且排在客訴偵測之前**（`line.js:1820-1824` vs `:2140`）：「上次送的菜壞掉了，訂單要退」含「訂」→ 走收單解析建單，客訴偵測永不執行、管理員無推播。

- **M16. rebuild 端 dedupe 與「連傳兩則相同文字＝兩行累加」語意衝突**（`line.js:726` 註解 vs `order-parsed-heuristics.js:39-56` 全欄相同即砍）：客戶同日兩則一模一樣的叫貨/重傳同一張照片，結單前是兩份、rebuild 後剩一份 → 可能少出貨。
- **M17. 失敗→LINE redelivery 重跑時 raw 附加不冪等**（`skipIfPresent` 只在接手逾時租約時啟用，`line.js:1293`；失敗路徑 finally 刪佔位列 `:2499-2503` 後重跑 `lineMessageIsRetry=false`）：第一次執行在 append raw 之後才失敗，重跑會把同段文字/OCR 再附加一次；Gemini 若解讀成加總數量會翻倍。

### 安全性
