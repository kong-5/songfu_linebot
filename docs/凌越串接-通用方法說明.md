# 凌越 ERP 串接 — 通用方法說明 / Playbook

> 這份是「把雲端後台的資料寫進凌越（或從凌越回讀）」的**通用做法**。
> 第一個落地案例是「訂單回寫凌越」（見 `docs/凌越回寫-工作交接.md`、`docs/凌越訂單回寫-串接說明.md`）。
> 要接**新的資料／新的單別**（如：銷貨訂單、進貨單、庫存盤點…）時，照這份的步驟與檢查表走即可。
> 最後更新：2026-06-09

---

## 0. 一句話原理

凌越是**公司內網（LAN）的 SOAP/WCF 系統**，雲端（Cloud Run）連不到內網。
所以一律拆成兩段，靠**一把共用金鑰**串起來：

```
 雲端 Cloud Run（外網，看得到資料庫）          公司 Windows（內網，連得到凌越 LAN）
 ─────────────────────────────────          ──────────────────────────────────
   後台出「待處理資料」API  ◀───── HTTPS 拉取 ─────  橋接程式 bridge.py
   後台收「凌越回傳結果」API ◀──── HTTPS 回填 ─────       │ 用 ly_datain.py 寫/讀凌越
              ▲                                          ▼
              └────── 同一把金鑰（兩邊各貼一次）────────  凌越 SOAP (LAN 192.168.4.x)
```

- **雲端只負責「出資料／收結果」**，永遠不直接碰凌越。
- **真正讀寫凌越的只有公司 Windows 那台**（有 `ly_datain.py`、連得到 LAN）。
- 兩台**不共用檔案**，唯一要手動同步的是**金鑰那串字**。

## 1. 角色分工（誰做什麼）

| 角色 | 在哪 | 負責 |
|---|---|---|
| **雲端後台** | Cloud Run `songfu-line-bot`（見 [memory] 部署服務名） | 寫兩支機器對機器 API（出資料 / 收結果），用金鑰驗證 |
| **橋接程式** | 公司 Windows `D:\Work\lystk_tool\` | 拉雲端資料 → 組成凌越要的 row → 呼叫 `ly_datain` 讀寫凌越 → 回填結果 |
| **`ly_datain.py`** | 公司 Windows（**Dispatch 維護，我們看不到**） | 真正封裝凌越 SOAP：`write_invoice` / `verify_sp_no` / `delete_invoice`…等 |
| **凌越 ERP** | 公司 LAN `192.168.4.x` | 最終資料落地 |

> **關鍵分界**：欄位名稱、資料種類代碼、單別、是否自動帶價 — 這些都是**凌越/`ly_datain` 的領域**，
> 由 Dispatch 提供。雲端與橋接程式不要憑空猜凌越 schema（猜錯會寫壞正式機）。

## 2. 通用步驟（接任何新資料都照這個順序）

### 步驟 1 — 雲端側：寫兩支機器對機器 API

放在 admin router（`dist/admin/index.js`），**不走 cookie 登入，改用 API 金鑰標頭**。
以「回寫」為例，命名慣例 `/admin/<功能>-writeback/...`：

- `GET /admin/<功能>/pending?date=YYYY-MM-DD` → 回「待處理」資料 JSON（通用欄位，不綁單別）
- `POST /admin/<功能>/callback` → 收凌越回傳的結果（如單據號），回填 DB

**認證樣板**（加在 `router.use` 最前面，比 cookie 驗證更早攔截）：

```js
router.use(async (req, res, next) => {
    const pathname = req.path || "/";
    if (pathname.startsWith("/<功能>/")) {
        const expected = String(process.env.<KEY_ENV> || "").trim();
        const provided = String(req.headers["x-writeback-key"] || "").trim();
        if (!expected) { res.status(503).json({ error: "<KEY_ENV> 未設定" }); return; } // 停用
        if (!provided || provided !== expected) { res.status(401).json({ error: "unauthorized" }); return; }
        req.adminUsername = "<功能>-agent";
        return next();
    }
    // …原本的 cookie 驗證…
});
```

回傳的 JSON **只給通用欄位**（客戶碼、料號、數量、單位、備註、日期…），
**不要在雲端綁死凌越單別** — 同一份資料可餵給銷貨單、銷貨訂單、進貨單等不同單別。

> DB 若要記「已處理」狀態，加兩欄就好：`<功能>_doc_no`（凌越配發單號）、`<功能>_written_at`（處理時間）。
> SQLite 與 Postgres 兩邊遷移都要加（`dist/db/index.js` 的 `initSqlite` 的 alters 陣列 + `initPg`）。

### 步驟 2 — 設金鑰（兩邊一字不差）

雲端（**只加這一把，不要動到其他環境變數**）：

```bash
gcloud run services update songfu-line-bot --region asia-east1 \
  --update-env-vars "<KEY_ENV>=<金鑰>"
```

> ⚠️ 不要用 `npm run deploy` 不帶 `--keep-env` 來設金鑰 — 那會用 `.env` 全量取代雲端環境變數
> （見 [memory] 部署 footgun）。設單一變數一律用 `gcloud run services update --update-env-vars`。

產生強金鑰：`echo "lywb_$(openssl rand -hex 24)"`。把同一把貼到 Windows 的環境變數。

### 步驟 3 — 部署雲端程式

```bash
node scripts/deploy-cloud-run.mjs --keep-env   # 只換映像，不動環境變數/Secret
```

部署的是**當前簽出的 `dist/`**。先確認在正確分支（見 [memory] 多分支並行），且程式是純增量、
`node --check dist/admin/index.js` 過、無衝突標記。

### 步驟 4 — 內網橋接程式（`scripts/<功能>_bridge.py`）

骨架（參考 `scripts/ly_writeback_bridge.py`）：拉 → 組 → 寫 → 回填，**四種模式**：

| 模式 | 行為 | 用途 |
|---|---|---|
| `--dry-run` | 只拉資料、組好 row、印出來，**不寫凌越** | 確認欄位對映正確 |
| `--test` | 只寫**第一張**（備註標記 `【API測試請刪除】`）→ 回查 → 自動刪 | 驗證凌越收不收 |
| `--test --keep` | 同上但**不刪**，留著自己進凌越看 | 怕出錯、要人工核對時用 |
| （不帶旗標） | **正式整批**，逐張寫、單張失敗不中斷、最後回填 | 確認無誤後上線 |

橋接程式靠 `X-Writeback-Key` 標頭打雲端，靠 `import ly_datain` 寫凌越。
HTTP 用標準庫 `urllib` 即可（免裝 requests）。

### 步驟 5 — 欄位對映（凌越領域，**跟 Dispatch 要**）

每個單別的欄位前綴、資料種類代碼**都不一樣**。接新單別前先問 Dispatch：

1. `ly_datain.py` 有沒有寫「該單別」的方法？函式名稱、參數？
2. 該單別在 LyDataIn 的**資料種類代碼**？（見下方對照表）
3. 主表 / 明細的**欄位名稱**？
4. **單價**留空不送，凌越會不會自動帶（客戶售價表 / 進貨價）？還是必須送？
5. **倉別**這單別需不需要？可不可以空？

### 步驟 6 — 驗證流程（每次接新單別都重跑一遍）

```
--dry-run            → 看組出來的客戶/料號/數量對不對
   ↓
--test --keep        → 寫一張，進凌越人工核對：單別對不對、欄位對不對、單價有沒有自動帶
   ↓ （確認無誤、手動刪掉測試單）
正式整批              → 回後台看資料出現凌越單號
   ↓
掛 Windows 工作排程器  → 每日自動跑（用 .bat 包裝，開始位置設 D:\Work\lystk_tool）
```

雲端端點本身可直接 curl 驗證：

```bash
# 無金鑰 → 預期 401 unauthorized
curl -s -o /dev/null -w "%{http_code}\n" "<URL>/admin/<功能>/pending"
# 帶金鑰 → 預期 200 + JSON
curl -s -H "X-Writeback-Key: <金鑰>" "<URL>/admin/<功能>/pending?date=2026-06-09"
```

## 3. 凌越資料種類代碼對照（已知）

| 代碼 | 種類 | 主表/明細前綴（已知者） |
|---|---|---|
| `000000` | 貨品主檔 | — |
| `000001` | 客戶主檔 | `CT_NO / CT_NAME / CT_UNINO / CT_ADDR1` |
| `000004` | 倉庫 | — |
| `000009` | 庫存 | — |
| `0000A0` | 訂貨 | — |
| `0000A1` | 銷貨（銷貨單） | 主 `SP_CTNO/SP_CTNAME/SP_DATE/SP_REM`；明細 `SD_SKNO/SD_NAME/SD_UNIT/SD_QTY/SD_PRICE/SD_WHNO/SD_REM` |
| `0000A2` | 銷退 | — |
| ？ | **銷貨訂單（客戶訂單）** | **待 Dispatch 提供**（這是「訂單回寫」要寫的目標） |

> WCF 函式：`LyGetPassKey`（取 30 秒金鑰）/ `LyDataIn`（寫）/ `LyDataOut`（讀）/ `LyDataPage` / `LyDataDel`。
> 認證流程與技術文件見 `docs/凌越API技術文件/`（在 HACCP_Project 主專案）。

## 4. 常見坑（踩過的，務必先看）

1. **「寫成功但凌越找不到」** → 九成是**查詢條件對不上**，不是資料不見：
   - **公司別 ICPNO**：橋接預設 `00`（松富）。凌越畫面若看別家公司就看不到。
   - **單別**：寫的是哪種單，就要去那個單別畫面找（銷貨單 ≠ 銷貨訂單 ≠ 訂貨）。
   - **日期**：用單據日期（`SP_DATE`）當查詢區間。
   - **審核狀態**：新寫入多半 `SP_CHECK=0`（未審核），很多查詢畫面預設只顯示已審核 → 要勾「含未審核」。
   - **最快**：用凌越單號或備註「`【API測試請刪除】`」直接搜。
2. **單價變 0** → 不要送 `SD_PRICE=0`（那是強制蓋成 0）。**留空＝不送該欄**，讓凌越依客戶售價表自動帶。
3. **倉別被拒** → 若凌越不收空倉別，設 `LY_DEFAULT_WHNO=<實際倉別>` 再試。
4. **撞單號** → 不要在雲端指定單號。讓凌越/`ly_datain` 依「當日既有單據最大流水 +1」自動順編。
5. **重複跑安全** → 已處理的資料（`<功能>_written_at` 非空）不會再出現在 pending，不會重複寫。
6. **金鑰兩邊不一致** → Cloud Run 的 `<KEY_ENV>` 必須 = Windows 的環境變數，一字不差。
7. **部署洗掉環境變數** → 設單一變數用 `--update-env-vars`；整批部署一律 `--keep-env`。

## 5. 安全與邊界

- **不要在沒拿到憑證下盲打凌越正式機**；新單別一律先 `--test --keep` 一張，人工核對後再正式。
- **凌越 schema（欄位/代碼/是否自動帶價）一律以 Dispatch 為準**，雲端與橋接不猜。
- **金鑰**只存在 Cloud Run 環境變數與 Windows 環境變數，**不要寫進 git、不要寫進 memory/筆記**。
- 機器對機器端點走 `X-Writeback-Key`，**未設金鑰回 503＝整個功能停用**（安全預設）。
