# 庫存管理 — 目前庫存（凌越 SK_NOWQTY）

後台「庫存管理 → 目前庫存」顯示凌越每個品項的**目前在庫量**，可每日自動刷新、也可點按鈕即時刷新，
支援列表檢視與「依倉庫」分組（凌越倉別 / 自訂庫房兩種），畫面緊湊、可搜尋、可匯出目前檢視。
最後更新：2026-07-17

> 架構與「不要重複踩」的權威清單在 repo 根目錄 **`CLAUDE.md`**；本文件為庫存細節。

---

## 1. 資料是什麼

- 來源＝凌越貨品主檔（資料種類 `000000`）的 **`SK_NOWQTY`（現有量）**，每品項一個「公司總量」。
  對照：料號 `SK_NO`、品名 `SK_NAME`、規格 `SK_SPEC`、單位 `SK_UNIT`、預設入庫倉別 `SK_RKWHNO`、
  **停用碼 `SK_STOP`**。
- **停用品（`SK_STOP=1`）不推上雲端**（推送時過濾），所以庫存與盤點都不會出現。每次推送依凌越
  當下狀態重新過濾，停用/復用下次推送自動反映，免人工維護。
- **負值是正常的**：出貨/銷貨先入帳、進貨還沒補時，帳面現有量會是負數。
- 「依倉庫」只是把品項**歸到某個倉來分組**，不是把數量拆到各倉（凌越給的是單一總量）。
  - **凌越倉別**：用 `SK_RKWHNO`（如 FN005/Y99）分組，免設定。後台盤點/倉庫設定**只認凌越倉號**。
  - **自訂庫房**：用後台「盤點作業（舊版）→ 品項歸倉」的設定分組；未歸倉者列在「（未歸倉）」。

## 2. 架構（雲端連不到凌越 LAN，一律兩段）

```
 內網 Windows（連得到凌越）                         雲端後台 Cloud Run
 ─────────────────────────                         ──────────────────
  ly_stock_push.py  撈 SK_NOWQTY ──POST 整批──▶  /admin/lingyue-writeback/inventory-push
  ly_stock_agent.py long-poll   ──等刷新請求──▶  /admin/lingyue-writeback/inventory-wait
                                                    ▼ 存入 erp_stock_items（全表覆蓋）
                                 使用者看 ◀──────  /admin/inventory/stock（庫存管理→目前庫存）
```

- 機器端點沿用**訂單回寫同一把金鑰** `LINGYUE_WRITEBACK_KEY`（標頭 `X-Writeback-Key`），不必另設。
- 「庫存更新」按鈕：後台記一個待處理旗標 → 常駐的 `ly_stock_agent.py` long-poll 立刻撿走 → 撈凌越 → 推回 → 頁面自動重載。
- **依公司自主更新（2026-07-17）**：刷新旗標可帶 `icpno`（`erp_stock_refresh_icpno`）。
  「目前庫存」頁按鈕＝只重推**當頁公司**；每日盤點頁的「更新最新庫存」旁有公司下拉（預設全公司）。
  `inventory-wait` 回 `{refresh:true, icpno}`，代理只重推該公司——**免改整合代理 LY_ICPNO 設定即可換公司更新**。

### 未來銷貨加回（2026-07-17）

有時「今天先打明天的銷貨單」進凌越，SK_NOWQTY 立刻被扣、但貨還沒出 → 快照低於實際在架量。

- 內網 `ly_stock_push.py` 推送時順帶查**單據日期＝明天～+`LY_FUTURE_DAYS` 天（預設 60）**的
  銷貨單 A1（加回）與銷退單 A2（扣回），逐料號淨量放 payload 頂層 `future_sales`。
- 雲端存 `erp_future_sales`（按公司覆蓋；查詢失敗不帶欄位＝保留上一份；帶空陣列＝清空該公司加回）。
- 後台「目前庫存」頁工具列有 **「未來銷貨加回」開關**（`.sf-switch`，全域設定
  `app_settings.stock_future_reversal_enabled`）：
  - **開**＝顯示庫存＝凌越快照＋人工調整＋未來銷貨淨量，數字旁藍 badge「未來+N」。
  - **關**＝遮蔽加回（回原凌越量、badge 轉灰提示），方便對照凌越原始數字查詢。
  - 只影響顯示，**不寫回凌越**、不影響每日盤點的「最新系統／盤差」計算（那套維持凌越＋人工調整）。

## 3. 內網那台要跑什麼（架構定案：統一管理介面）

**內網以「凌越整合代理」視窗程式 `ly_agent_gui.py` 為唯一常駐入口**，一個視窗同時掛庫存推送與
訂單回寫。庫存推送邏輯在**該資料夾內的 `ly_stock_push.py`**（GUI 用 `local_import` 優先載入它，
蓋過 `D:\Work\lystk_tool` 舊版）；底層 `lystk.py` 才在 `D:\Work\lystk_tool`。

- 推送時 `SK_STOP=1` 的停用品會被跳過，記錄框印 `（已跳過停用品 N 項 SK_STOP=1）`。
- 定時推送時間由 `LY_STOCK_TIMES`（如 `06:00,12:00`）控制；清空＝只靠按鈕/即時。
- 「庫存更新」按鈕：後台記待處理旗標 → GUI 的庫存執行緒 long-poll `inventory-wait` 撿走 → 撈凌越 → 推回。

**更新庫存模組 SOP**（拿到新版 `ly_stock_push.py`）：
- 跑 `.py`：新檔丟進「凌越整合代理」資料夾蓋掉舊的 → 重開視窗 → 按「立即推庫存」。
- 跑打包 `.exe`：新檔放 `凌越整合代理.exe` **同一層**（`local_import` 會優先讀它，**不必重打包**）→ 重開。
- ⚠ 不要放到 `D:\Work\lystk_tool`——GUI 不會用那裡的版本。權威說明見該資料夾 `規則與必備設定.md` §9。

> 舊的獨立跑法（`ly_stock_agent.py` / `ly_stock_daily.bat`）**已於 2026-07-14 刪除**——它會優先載到
> `D:\Work\lystk_tool` 的舊版推送邏輯（與 GUI 的「同層優先」相反），有拿錯版本的風險。一律開整合視窗。

環境變數（視窗⚙設定或環境變數）：

```
LY_CLOUD_BASE=https://<後台網址>
LINGYUE_WRITEBACK_KEY=<與後台相同那把>
LY_ICPNO=00          # 松富；01=龍港、03=桂田
LY_STOCK_TIMES=06:00,12:00   # 每天自動推送時間；清空＝只靠按鈕/即時
```

## 3b. 盤點頁的單位

盤點頁（LIFF）系統量與盤點輸入框旁會顯示**單位 `SK_UNIT`**（如 KG/把），規格 `SK_SPEC`
（如 18KG/箱）顯示在品名下。單位資料一直隨庫存推送（`SK_UNIT`），不需另抓。

---

## 4. 雲端只需確認一件事

`LINGYUE_WRITEBACK_KEY` 已設（訂單回寫在用就已經有）。沒有的話：

```bash
gcloud run services update songfu-line-bot --region asia-east1 \
  --update-env-vars "LINGYUE_WRITEBACK_KEY=<金鑰>"
```

資料表 `erp_stock_items` 由程式啟動時自動建立（SQLite/Postgres 皆有），免手動遷移。

## 5. 驗證

```bash
# 內網：先 dry-run 看撈得到幾筆（會一併印 000009 分倉樣本＋未來銷貨 A1−A2 淨量，核對「先打明天的單」有沒有被抓到）
python ly_stock_push.py --dry-run
# 內網：正式推一次
python ly_stock_push.py
# 雲端：打開「庫存管理 → 目前庫存」應看到品項與「資料時間」；按「庫存更新」→ 需 agent 常駐才會即時刷新
# 未來銷貨加回：推送成功後開頁面右上開關，或直接看有未來單的品項是否出現「未來+N」badge
```

**部署未來銷貨加回／依公司更新（內網側）**：把新版 `ly_stock_push.py` 與 `ly_agent_gui.py`
丟進「凌越整合代理」資料夾（exe 版放 exe 同層即可，`local_import` 會優先撿走、免重打包）→ 重開視窗。
雲端側推 `main` 自動部署，`erp_future_sales` 表由 DB init 自動建立。舊代理配新後台、或新代理配舊後台
都相容（欄位／icpno 缺省時各自回退舊行為）。
